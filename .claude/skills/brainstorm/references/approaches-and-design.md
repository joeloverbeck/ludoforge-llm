# Approaches and Design Presentation

Covers Step 3 (propose 2-4 approaches with tradeoffs and recommendation) and Step 4 (present the chosen design section by section). Both always-loaded.

## Step 3: Propose Approaches

Present **2-4 distinct approaches** with:

- **Name**: A short descriptive label
- **How it works**: 2-4 sentences
- **Tradeoffs**: What you gain, what you give up
- **Recommendation**: Lead with your recommended option and explain why

A 4th approach is justified only when it sits on an axis orthogonal to the primary 3 (i.e., not just a variant intensity of the same idea) AND the reference file documents it as load-bearing context that materially changes the recommendation. Default to 2-3; the 4th is opt-in, not encouraged.

**Worked example — orthogonal vs variant intensity**: "profile-only", "phased witness+optimization", and "two-spec split" are NOT three orthogonal axes — they are three commitment-scope intensities of the same axis (how much to commit before measuring). Treat as one approach with sub-options unless one of them carries a load-bearing distinction the others lack. By contrast, "skip the witness and commit directly to the architectural change" IS orthogonal to that axis (different commitment *shape* entirely, not a different intensity), so a 4th approach is justified. The orthogonality test: would picking approach A vs B change *what* the next step looks like, or only *how big* it is? Only the former counts as a separate axis.

**Mixed structural + intensity sets**: When candidate approaches mix structural variation (different artifact shape, different sequencing topology — e.g., "one spec" vs "two parallel specs") with intensity variation within one structure (more or less scope inside the same shape — e.g., "one phased spec" vs "one umbrella spec"), structural variations are orthogonal axes that each warrant their own approach; intensity sub-variations within one structural choice should be collapsed into sub-options of that approach. Example: "one phased spec" and "one umbrella spec" are intensity variations of single-spec structure (collapse into one approach with sub-options); "one spec" vs "two parallel specs" is structural variation (separate approaches).

**If the reference file already contains evaluated approaches** with tradeoffs and counter-evidence, present those as the approach options rather than generating new ones. The brainstorm's value in this case is validation and decision, not ideation. You may add a new approach if the reference file's options have a clear gap.

**If the user's invocation directly specifies the artifacts to produce** (e.g., "draft specs 163 and 164", "rewrite this skill", "create these N tickets"), the approach-level question is foreclosed by the user's instruction. This forecloses the artifact *type*, not the design choices within it: when the reference contains evaluated approaches concerning the artifact's internal design (e.g., which fix option), the "If the reference file already contains evaluated approaches" rule above still governs that choice — present them recommendation-led per the Decision-requiring-design "decide which to keep" path, not as a bare framing recap. Step 3 collapses to a one-paragraph framing recap (what artifacts will be produced and why this scope) plus any cross-cutting decisions (e.g., write independently vs. merge, sequencing, scope partitioning). Do not synthesize alternative approaches the user has already eliminated. The Step 2 compound-move still applies; the gap-closer batches the cross-cutting decisions, not synthetic alternatives. Because the Step 2 compound-move still applies, the Step 5 Section-preview gate fires — send the section-bullet preview before the Write call.

**Slicing decision when source recommends more artifacts than the user requested**: If the source artifact (external LLM proposal, prior spec's deferred-work section, internal report, etc.) recommends N artifacts and the user requests M < N — or an unspecified count that the source's enumeration exceeds (e.g., "create a spec" singular against a 5-spec roadmap) — the slicing decision is one of the cross-cutting decisions to surface in the compound-move gap-closer. Enumerate which N-K artifacts will be produced now vs. deferred to follow-up, with deferral rationales (dependency ordering, complexity, risk, scope, source's own sequencing guidance), so the user can correct the slicing before the artifact write. The Reassessment section in the produced artifact (see Step 5) MUST record the deferred items by name and reason so traceability survives the slicing decision — without that record, the deferred items become invisible to whoever decomposes or implements the produced artifact.

**If triage produced a set of approved changes** (decision/triage → design transition), the approach options shift from "which changes" to "how to apply them" — e.g., incremental patches vs. structured rewrite vs. phased rollout. Present these implementation strategies as the approaches.

**If implementation-related**: For each approach, note which FOUNDATIONS.md principles it aligns with or tensions it creates. Use format: `Foundations: Foundation #1 (aligns), Foundation #8 (tensions — [reason])`. Omit the line for an approach that is FOUNDATIONS-neutral relative to its alternatives — only tag when it surfaces a real differentiator (alignment unique to this approach, or a tension absent in the others). When all approaches are FOUNDATIONS-equivalent, defer per-approach tagging entirely and address FOUNDATIONS in the Step 4 design section.

**If an approach creates a new artifact dependent on an existing in-flight artifact**: state the dependency direction explicitly in the approach's "How it works" — e.g., "Spec 180 depends on Spec 179 landing first because the witness needs 179's signal." This prevents the user having to ask about ordering after approval. The dependency direction should appear in the approach text, not only in the produced artifact's Dependencies field — the user reviews the approach before the artifact exists. This rule fires most commonly under the Conditionally-foreclosed in-between path (Step 1.2) and the Slicing-decision path (above) where the recommended approach creates a follow-up artifact rather than replacing the existing one.

**Tactical + strategic compound**: When the user's pain has both a short-term unblock and a strategic root-cause fix, present them on two orthogonal axes — strategic approaches A/B/C with their own tradeoffs, plus an orthogonal tactical option T whose effort and risk are independent of the strategic axis. The compound gap-closer in Step 2 then offers strategic × tactical combinations (e.g., `A only`, `B only`, `B + T`, `C + T`) rather than mutually-exclusive choices. **Triggers**: use this pattern when the user's request explicitly references chronic pain plus a strategic horizon, or when CI/operational state is broken in a way that a tactical config change would unblock independently of the architectural fix. **Anti-triggers**: skip the tactical option entirely when all three of (a) the strategic complexity is Medium or smaller (weeks, not months), (b) the pain is degraded ergonomics rather than a blocked workflow (slow CI is not blocked CI), and (c) the tactical change requires non-trivial revert work whose cost approaches the strategic-only cost hold. The default posture is strategic-only — tactical is opt-in and conditional on positive triggers, not on the absence of objections; the strong "MUST come paired" enforcement below applies only after triggers fire, never as a license to default to tactical. Tactical options MUST come paired with a restoration ticket that closes only when the strategic phases land — without the restoration ticket, the tactical option silently normalizes as "the answer" and violates Foundation #15 (Architectural Completeness).

**Wait for user to choose or ask questions.** Do not proceed until the user picks an approach (or asks you to refine/combine).

**If your own follow-up evidence contradicts an earlier compound-move presentation** (e.g., a long-running measurement that completes after the initial recommendation, fresh investigation findings from a multi-stage Step 1.5/2 sweep, downstream tool output that reveals new constraints), present a revised compound-move message that (a) explicitly labels the revision as evidence-driven (e.g., "this inverts what I said earlier"), (b) explains what new evidence triggered the revision, (c) re-presents the approaches with updated framing — preserve approach labels where the underlying option is unchanged so the user can correlate, relabel only when an approach is genuinely new. Do not silently swap approaches without acknowledging the prior recommendation was based on incomplete data. This is the self-driven counterpart to the user-driven rejection rule below — both are evidence-driven iterations on the solution space, not confidence regressions.

**If the user rejects all approaches** or asks for re-analysis, investigate the concerns raised, then present a revised set. This is not a confidence regression — it's an iteration on the solution space. Do not restart the interview unless the rejection reveals a misunderstanding of the problem.

## Step 4: Present Design

Once an approach is chosen, present the design **section by section**. Scale each section to its complexity — a sentence for trivial parts, up to 200 words for nuanced parts. For designs with fewer than 3 substantive sections, present the full design in a single message and ask for overall approval rather than section-by-section. Reserve section-by-section flow for designs with 4+ sections where individual sections warrant independent review.

Sections to cover (skip irrelevant ones):

1. **Overview**: What this design achieves in 1-2 sentences
2. **Architecture / Structure**: How the pieces fit together
3. **Key decisions**: Important choices and why
4. **Data flow / Process**: How information moves through the system
5. **Edge cases**: Known tricky scenarios and how they're handled
6. **Testing strategy**: How to verify this works (if implementation-related)
7. **FOUNDATIONS.md alignment**: Table of relevant principles and how the design respects them (if implementation-related)
8. **Decision Tree** (for investigation/research specs and diagnostic-mode reports): A table or structured prose mapping per-hypothesis or per-phase findings to action outcomes (e.g., "if H1 dominates: Accelerate via follow-up spec X; if H2 dominates: Keep-as-correctness-only; if no single dominant cause: Retire"). Used when the spec's deliverable is a decision report and the implementer (or the diagnostic-mode synthesis phase) must commit to one branch based on measured evidence. Distinct from "Key decisions" — that section records decisions already made; Decision Tree records branching decisions deferred to phase outcomes.

The list above is a starting menu, not a fixed schema — domain-appropriate substitutions (e.g., "Phases" or "Step-by-step execution" in place of "Data flow / Process") are expected for non-implementation designs.

**Repo-convention precedence**: For implementation specs targeting a repo with an established spec convention (visible by inspecting `specs/` and `archive/specs/`), follow that convention's section grammar instead of the menu above. The menu is a fallback for non-implementation designs and for repos without an established convention. The repo's own grammar (e.g., Goal / Non-goals / Foundation amendments / Architecture / Phases & acceptance criteria / Test plan / Foundation alignment table / Out of scope / Open questions, as visible in this repo's `archive/specs/`) is the source of truth when present.

"Implementation-related" means the design will result in changes to source code governed by FOUNDATIONS.md. Skill design, process changes, and tooling configurations are not implementation-related for this purpose, even if they indirectly influence implementation.

**After each section**, ask: "Does this section look right?" Wait for confirmation before presenting the next section. If the user pushes back, revise that section before continuing.

**Auto-mode adaptation**: When Claude Code's auto mode is active, section-by-section gating compresses to consolidated presentation with a single approval. Present the remaining sections together and proceed to Step 5 artifact writing unless the user has pushed back on a prior section. This matches auto mode's "prefer action over planning" posture while preserving substantive review — the user still sees every section before the artifact is written. If a user objection arises mid-consolidation, stop, revise the flagged section, and present the revision for approval before continuing. When "no clarifying questions" is the active directive (a softer constraint than auto-mode UI, but functionally equivalent for Step 4 compression purposes), the same consolidated-presentation behavior applies — see Pre-Set Directives §4 (in `references/interview-protocol.md`) for the section-preview gate that accompanies the compression.

**Compound-move + auto-mode intersection**: When Step 2's compound-move variant was used (approach presentation merged with terminal gap-closer), the consolidated Step 4 design preview is still required as a separate message before the artifact write — section-name bullets with a one-line summary each are sufficient, full prose is not. The compound-move's approach-level overview does NOT satisfy the section preview promise, since later sections (edge cases, recovery info, files NOT touched, step-by-step execution) are typically not enumerated at approach time. Skip the preview only when the user has explicitly waived it. Phrasings that DO count as preview waiver: "just write the file", "skip the preview", "no need to walk me through it", "go ahead and write". Phrasings that DO NOT count: "proceed with option X" (approves approach, not preview), "without stopping for clarifying questions" (constrains interview rounds, not transparency gates), "continue" / "go" (ambiguous — treat as approach approval only). **Default for unenumerated phrasings** (e.g., "go with recommended", "proceed with X" / "proceed with your recommendation" / "proceed with the plan", "looks good", "sounds right", "do it"): treat as approach-approval-only, not preview waiver — send the section-bullet preview before the Write call. The "proceed" family extends the explicit "proceed with option X" entry from the DOES NOT count list above — every "proceed with ..." phrasing is approach-approval-only unless paired with an explicit waiver phrase from the DOES count list. When in doubt, preview as a separate message — the user can always say "skip the preview" the next time around. Per Pre-Set Directives §4 (in `references/interview-protocol.md`), "separate message" means separate from the Write tool call only — bundling section-name bullets in the same prose message as the approach presentation is permitted (and often efficient under "no clarifying questions" directives), provided the bundle lands before the Write call.
