# Writing Output Artifacts

Covers Step 5: writing output artifacts for each mode (design / decision-triage / operational / multi-file directory), the section-preview gate, the numbering and replacement conventions, the Reassessment-section requirement for external-LLM outputs, phased-spec acceptance budgets, destructive-action sections, and the Design Dismiss outcome path. Always-loaded.

## Step 5: Write Output Artifacts

**Plan mode override**: If plan mode is active, the harness specifies the artifact path; write there instead of the per-mode default below. See `references/plan-mode.md`.

**Section-preview gate**: If Step 2 used the compound-move variant and the most recent user-authored input (the original request or any subsequent message) did not contain an explicit waiver phrase from the Step 4 enumeration ("just write the file", "skip the preview", "no need to walk me through it", "go ahead and write"), ensure the section-bullet preview lands in an assistant message earlier than the one that makes the first Write tool call (or, when the deliverable is in-place edits to existing files, the first Edit call) — one preview covering all planned files in a multi-file output satisfies the gate regardless of how the subsequent Writes are message-batched; bundled with the approach recap in that prior message is fine; what is not permitted is putting the bullets in the same assistant message as the Write call (whether inside the call or as prose preceding it in that same message), since that gives the user no turn in which to object before the file is written. The canonical definition of "before the Write call" (vs. "as a standalone dedicated message") is in Pre-Set Directives §4 in `references/interview-protocol.md`. This applies even under "no clarifying questions" (which constrains interview rounds, not transparency gates per Pre-Set Directives §4). Zero-round interviews — under "no clarifying questions", High-Confidence Start with no intervening user turn, or any flow where the section preview message is the first prose since the original request — do NOT waive the gate; the absence of an intervening user response is not a waiver, and only an explicit waiver phrase in the user's prior input qualifies. This gate also fires on the Step 3 foreclosed path (user's invocation directly specifies the artifacts) because the Step 2 compound-move still applies there per Step 3's foreclosure paragraph; the absence of an approach presentation does NOT exempt the artifact write from the section-bullet preview. Skip this gate when plan mode is active — the plan file is itself the preview surface. After the Write call lands, proceed to Step 6 — the next-steps menu (or auto-mode recommendation under "no clarifying questions") is still required; Step 5 is not the terminal step.

**Procedural counterpart — yield turn after preview**: The gate above describes the ordering invariant ("preview in an earlier message than the Write"); the corresponding procedural action is *end the assistant turn after the section-bullet preview*. Concretely: when the section-bullet preview message is composed, it MUST be the last content in that assistant response — no Write or Edit tool call in the same response, no "now writing" sign-off followed immediately by tool calls in the same turn. The next assistant turn — triggered by the user's response (object, redirect, or affirm) under standard mode, or by an auto-mode tick after the user has a chance to object under "no clarifying questions" / auto mode — is the one that issues the first Write call. This is the most common section-preview-gate violation under auto-mode: the natural disposition to "keep going" lets an assistant satisfy "preview before Write call *ordering-wise*" while collapsing both into one response and depriving the user of the objection turn the gate exists to create. Treat the gate as a turn-yield constraint, not just an ordering constraint.

**Numbering convention (applies to spec/ticket outputs)**: When writing specs or tickets, determine the next available number by inspecting existing files. Directory inspection of `specs/`, `archive/specs/`, `tickets/`, and `archive/tickets/` is sufficient when the repo's archival workflow is current (the common case — completed specs/tickets land in `archive/` predictably). The git-history grep (`git log --oneline --all | grep -oP '[Ss]pec \K[0-9]+'`) is the defensive variant for repos where unarchived in-flight specs may exist only in commit history; use it when directory inspection turns up gaps in the numbering sequence or when the most recent merges have introduced specs not yet visible in `archive/`. For tickets continuing an existing prefix series (e.g., `POLPREVDRIVE-001` is archived; new tickets are `POLPREVDRIVE-002+`), inspect `archive/tickets/<PREFIX>-*.md` for the highest existing number and continue from `+1`. Follow established formatting conventions from existing specs.

**Replacement convention (applies when superseding existing in-flight artifacts)**: When the user's invocation asks to remove, replace, or rewrite an existing in-flight artifact rather than create a fresh one (e.g., "remove this spec and create a new one", "rewrite this design"), choose the replacement path before writing and surface the choice in the Step 4 / Step 5 transition so the user can override the default (in a compound-move flow, where Step 4 is compressed, surface it in the compound-move gap-closer message or the section-bullet preview instead):

- **Same number, overwrite in place**: When the prior artifact is PROPOSED but never merged or decomposed (no tickets exist, no commits depend on its specific content), overwrite the file at the same path. Git history preserves the prior version; no archival action needed. This is the default when the prior version was authored earlier in the same session or is otherwise clearly transient.
- **Same number, archive prior version**: When the prior artifact was decomposed into in-flight tickets or merged into dependent artifacts, move the prior version to `archive/specs/<number>-<name>-superseded.md` (or the repo-equivalent archival path per `docs/archival-workflow.md`) before writing the new one, with a one-line header noting the supersession reason.
- **New number assigned (replacement)**: When the prior artifact reflects a fundamentally different scope decision (e.g., a coarse-grained spec being decomposed into two narrower specs whose union differs from the original), assign a new number and leave the prior artifact at its existing path; archival follows the repo's normal completion workflow.
- **Keep prior + new artifact (complement, not replace)**: When the user's conditional invocation resolves to partial adoption (Step 1.2 Conditionally-foreclosed in-between path) — the prior artifact is correct as authored AND a new follow-up artifact captures durable improvements not addressed by the prior — the prior artifact stays in place unchanged (no overwrite, no archival) and a new artifact with a new number is created. State explicitly in the Step 6 summary that the prior was kept unchanged (not archived), since the user's original binary likely anticipated archival on the rejection branch. Pair with Step 3's dependency-direction rule when the new artifact depends on the prior.

Do not archive silently — per `docs/archival-workflow.md` discipline, archival decisions are explicit. State the chosen replacement path in the Step 6 summary alongside any Step 1.5 verification-artifact disposition, so the user can object before next steps.

### Design mode (default)

Once all sections are approved, determine the output format:

- **If the design needs further refinement** (sections had significant revision, open questions remain that block the design from being implementable as-is, approach is exploratory): write to `docs/plans/YYYY-MM-DD-<topic>-design.md`. Include a "Brainstorm Context" header noting the original request, reference file (if any), key interview insights, and final confidence score with any assumptions. The standard "§11 Open questions" (or equivalent) section in repo-convention specs — documented follow-up tracking that does not block implementation — does NOT trigger this criterion; only design uncertainty that prevents the next implementer from proceeding does.
- **If all sections were approved without revision and the output is a well-scoped implementation spec** (ready for ticket decomposition): write directly to `specs/<number>-<name>.md`. The design doc is a staging area for designs that need further discussion — not a mandatory waypoint when the brainstorm produces a finished spec.
- **If the design implies a FOUNDATIONS.md amendment** (the existing principles don't cover a new construct's required behavior, or an existing principle needs extension): draft the amendment text as a section of the produced spec (typical pattern visible in recent repo specs like `archive/specs/{140,144,162,164,180}`, which amended Foundations #5/#10/#18/#19/#20). State the existing principle, the proposed change/extension, and the rationale. The amendment lands with the spec's implementation, not before; the user reviews both in one pass. Standalone FOUNDATIONS-only edits with no implementing spec are rare — when warranted, write the amendment as a `docs/plans/YYYY-MM-DD-foundation-<NN>-<descriptor>.md` proposal for review.
- **If the brainstorm produces a new user-invocable skill**: write to `.claude/skills/<name>/SKILL.md`. Follow the convention visible in sibling skills under `.claude/skills/` — frontmatter (`name`, `description`, `user-invocable: true`, optional `arguments` with `name`/`description`/`required`), worktree-awareness section, numbered Process steps, Guardrails. Use the multi-file directory pattern (`SKILL.md` + `references/`) only when SKILL.md would exceed ~250 lines or when distinct instruction surfaces warrant extraction; defer to `skill-extract-references` for retroactive splitting rather than pre-splitting at creation time.
- **If the brainstorm output is a measurement/decision report** (diagnostic mode per Step 1.2; gate decisions, witness reports, perf/regression analyses producing a verdict + named follow-up artifacts): write to `reports/<topic>-<descriptor>.md` matching existing repo convention (e.g., `reports/<spec-number>-phase-<letter>-<descriptor>-gate-decision.md` for gate decisions, or `reports/<topic>-<YYYY-MM-DD>-<descriptor>.md` for standalone investigations). Common when Step 1.5 verification is the bulk of the work and the deliverable answers a yes/no or quantitative question rather than designing implementation. The report MUST name follow-up artifacts (spec, ticket, "none — investigation closed", or an external/out-of-repo handoff — a deep-research brief, vendor ticket, or upstream issue, when the investigation continues outside this repo) so a future reader can navigate forward from the measurement to its operationalization; include a Decision Tree section (per Step 4 menu entry 8) when the verdict branches on findings rather than committing to a single outcome.

**Deep-research / external-LLM handoff brief**: When a diagnostic report's primary purpose is to be handed to an external researcher (a ChatGPT-Pro deep-research session, an upstream maintainer, a vendor) rather than decomposed into repo artifacts, it is still a `reports/<topic>-<descriptor>.md` output, but its grammar centers on: (a) a self-contained problem statement — the external reader lacks repo context; (b) verified findings with source anchors; (c) prior art already tried, so the researcher does not repeat it; (d) the precise question(s) for the researcher; (e) a source-anchor / artifact index for navigation. The "follow-up artifact" it names is the external handoff itself (the routing rule's external/out-of-repo case above), not a repo spec/ticket. Step 6 uses the external-handoff menu variant in `references/next-steps-menu.md`.

**Compound-move "approved without revision"**: In any flow where sections are previewed as bullets but never individually approved — the Step 4 auto-mode adaptation combined with the compound-move section preview, or the compound-move section preview used outside auto-mode (the High-Confidence Start variant at 80–94%) — the section-bullet preview without user pushback counts as "approved without revision" for the routing rule above. Route to `specs/` if the design is well-scoped, route to `docs/plans/` only if the user pushed back on a section in the preview message. Pushback that arrives AFTER the artifact has been written is post-design feedback (handled per "Post-Design Requests" in the thin SKILL.md), not a routing reclassification.

**Multi-spec interdependent output**: When the design yields 2+ *interdependent* specs in one pass, also write an ordering/index artifact (`specs/IMPLEMENTATION-ORDER.md`, or the repo-equivalent) capturing what no single spec owns: prerequisites already landed, the ordered series with a one-line per-spec rationale and explicit dependency direction, and any deferred/out-of-series items by name with deferral reasons. Scope this to the interdependent case only — independent parallel specs do not need an ordering index (avoid scope inflation). When the user names a literal index filename (e.g., `specs/IMPLEMENTATION-ORDER.md`) that conflicts with an established repo convention (e.g., dated `specs/IMPLEMENTATION-ORDER-YYYY-MM-DD.md` files visible in `specs/` / `archive/specs/`), honor the literal request by default but flag the convention divergence in the compound-move gap-closer or section-preview so the user can choose the dated form instead.

**Independent specs with user-requested ordering index**: When the user's invocation explicitly requests an ordering index for specs the source artifact recommends NOT ordering (independent specs), honor the user's instruction by producing a lightweight index that records the independence verdict as the central content — no dependency-direction line; specs listed as parallel items; per-spec one-line rationale captured from the source artifact's findings. Cross-reference the source artifact's opt-out rationale in the index so future readers can trace the decision. The index in this case is forward-navigation infrastructure for the user (a discoverable entry point to the related-but-independent spec set), not the ordering-arrow contract its name implies. This is the symmetric dual on the artifact-creation axis to the "Slicing decision when source recommends more artifacts than the user requested" rule in `references/approaches-and-design.md` — both rules resolve user-instruction-vs-source-recommendation tension by honoring the user's literal instruction while preserving traceability of the source's rationale.

**Reassessment section for source-derived outputs**: When the artifact is a Decision-requiring-design output derived from a source artifact whose recommendations the produced artifact selectively adopts — an external LLM proposal (per Step 1.2 "External LLM analysis"), OR an internal report whose Follow-Up Specs / Next Steps / Deferred-to-follow-up section names the produced artifact — include a final "Reassessment of source proposal" section enumerating per-recommendation dispositions: adopted (no change), adopted with adjustment (concept kept, implementation modified — with the change noted), corrected (with what was wrong — applies both to recommendation-level errors AND to stale citations, dependency claims, or framing where the architectural recommendation itself is retained), deferred (either to a named follow-up spec, or conditionally/uncommitted with no artifact yet — e.g., "revisit only if post-fix profiling still shows a hotspot"), rejected (with rationale), or meta-decision (response to a user question adjacent to the source proposal — e.g., "no FOUNDATIONS amendment warranted", "no scope expansion needed in dimension X" — used when the user's invocation poses standalone questions that don't have a source-proposal counterpart). This preserves traceability of the brainstorm's decisions through to the implementer who otherwise has no record of which external proposals were honored versus modified. When one proposal yields multiple specs, give each spec a Reassessment section scoped to *its* slice's dispositions, and record the cross-spec slicing rationale (deferred items by name) in the ordering/index artifact or the lowest-numbered spec, so per-slice traceability survives the decomposition.

**Closing section for diagnostic-mode outputs**: Diagnostic reports benefit from a structured closing section analogous to the source-derived Reassessment, but the framing is different — the closing records the measurement's scope and limitations, not per-recommendation dispositions. Include: (a) measurement environment caveats (host, runtime, software versions where they affect interpretation), (b) explicit scope boundaries (what was NOT measured and why, so a future reader knows when a follow-up diagnostic is warranted), (c) the named follow-up artifact (already required by the diagnostic routing rule above, echoed here for the section's completeness), (d) verification artifact disposition per the four-category list in `references/context-and-classification.md` §1.5, (e) replacement convention statement when the report supersedes a prior measurement. Title the section "Reassessment" or an equivalent closing label that fits the report's voice — the name is less important than the contents.

**Correcting an archived predecessor**: When the produced artifact reverses or corrects a claim in an *archived* spec/ticket (e.g., an open question the new artifact resolves differently), do NOT modify the archived artifact — archival is historical per `docs/archival-workflow.md`. Document the reversal in the new artifact's Context and Reassessment sections so traceability is preserved forward.

**Phased-spec acceptance budgets**: If the spec has phased delivery (Phase 0/1/2/... structure), include a per-phase acceptance row — either as a phase-boundaries table or as per-phase prose subsections — where each phase pairs a measurable criterion (latency budget, test pass rate, parity proof, etc.) with the phase's effort estimate. Tabular form is easier for `spec-to-tickets` to scan at a glance; per-phase prose subsections are acceptable when each phase warrants more narrative explanation than fits a table cell. Either shape works as long as each phase's measurable criterion and effort estimate are explicit. This is the primary scaffolding `spec-to-tickets` consumes when it decomposes the spec into ticket waves; without explicit per-phase budgets, decomposition becomes guesswork. Single-phase specs do not need this structure — a single acceptance criteria section suffices.

**Destructive-action sections**: If the design prescribes destructive or irreversible actions (file deletion, branch-protection edits, dependency changes, schema migrations, force-push, etc.), include the operational-mode sections — *Verified state*, *Step-by-step execution*, *Verification checklist*, *Recovery info*, and *Files NOT touched* — regardless of which output format above applies. These sections turn a design into a safe-to-execute plan and prevent the implementor from improvising recovery on the spot.

**Design Dismiss outcome**: When the brainstorm concludes that no artifact should be produced (e.g., the recommendation is "don't do this" after evaluating approaches), do not slide into silence. The dismissal is a legitimate brainstorm outcome that mirrors triage mode's `Dismiss outcome`, and the analytic work product needs an exit path:

1. Confirm the dismissal rationale with the user explicitly. Recap the corrected reasoning so the user can object before the thread closes.
2. Offer to write a short rationale memo to `reports/YYYY-MM-DD-<topic>-rationale.md` capturing the dismissal reasoning, including any corrections to prior specs, reports, or analyses. The memo is opt-in — if the user declines, end without recording.
3. If the dismissal surfaced a documentation gap in a prior spec, report, or skill (e.g., the prior artifact's reasoning is now miscalibrated by the brainstorm's findings), offer to update it (Continual Learning, per CLAUDE.md). Step 6's Continual Learning prompt is structurally bypassed by the no-menu path on dismissal, so it must fire here instead.
4. End. Do not present Step 6's next-steps menu — there is no artifact to act on.

Do NOT commit the file. Leave it for user review.

### Decision/triage mode

If the brainstorm's output is specs or tickets (not a design requiring further refinement), skip the design doc and write the artifacts directly:
- **Specs** go to `specs/<number>-<name>.md` following existing spec conventions
- **Tickets** go to `tickets/<PREFIX>-<NNN>-<name>.md` following `tickets/_TEMPLATE.md` for section structure

**Reassessment section for source-derived outputs (triage mode)**: If the source artifact qualifies per the Design-mode "Reassessment section for source-derived outputs" rule above (external LLM proposal per Step 1.2 OR internal decision-handoff report whose Follow-Up / Next Steps section names the produced artifact), apply that rule to the written spec or ticket — include a per-recommendation dispositions section so the implementer can trace which recommendations were adopted, modified, deferred, or rejected.

### In-place documentation edit (no new artifact)

When the brainstorm concludes that the warranted action is editing pre-existing documentation in place — decontaminating a `reports/` file, relabeling spec / `IMPLEMENTATION-ORDER` framing, correcting a cookbook entry — and produces no new spec/ticket/plan/report and is not a dismissal, the deliverable IS the set of edits. This is a common Decision-requiring-design / external-LLM outcome ("this is a documentation problem, not a code problem", per the Step 1.2 Conditionally-foreclosed in-between path in `references/context-and-classification.md`). Routing and gates:

- **Output path**: the edited files themselves — no new path is assigned. State the full set of edited files in the Step 6 recap so the user sees every touched file.
- **Section-preview gate**: applies to the Edit batch exactly as it applies to a Write. Land the per-file change-bullet preview in an assistant message earlier than the one that makes the first Edit call (per the Section-preview gate above, which covers in-place-edit deliverables).
- **Reassessment record**: when the source qualifies as a source-derived output (external LLM proposal per Step 1.2, OR an internal report whose Follow-Up / Next Steps section names the produced artifact), the per-recommendation dispositions still need a home, but there is no single produced artifact to host a "Reassessment of source proposal" section. Append the dispositions as a short reassessment/decision breadcrumb to the source proposal document itself (adopted / adopted-with-adjustment / corrected / deferred / rejected, per the Design-mode Reassessment rule above). This doubles as the Continual Learning supersession breadcrumb when the brainstorm's conclusions supersede the source's load-bearing recommendation.
- **Step 6**: there is no per-output-type menu — the flow is naturally complete. Collapse Step 6 to the Continual Learning offer (when it fires) plus a one-line recap of every edited file. Do not synthesize a spec/ticket/plan menu for an outcome that produced none.

Do NOT commit the edits. Leave them for user review.

### Operational mode

Write an executable plan with the following sections (scale each to its complexity):

- **Context**: Why the action is being taken — the problem, prompt, or intended outcome
- **Verified state**: Concrete observations from Step 1.5 (commit SHAs, file inventories, test results, etc.) so a reader can confirm the plan is grounded in current reality
- **Decisions**: Scope and risk decisions made during the interview, with rationale
- **Step-by-step execution**: Numbered steps with the exact commands or actions, expected outputs, and any conditional branches (e.g., "if dry-run reveals X, pause")
- **Verification checklist**: How to confirm the action succeeded (commands and expected results)
- **Recovery info**: How to undo if something goes wrong (where applicable — e.g., reflog, backup paths)
- **Files NOT touched**: Explicit list of paths/state intentionally outside scope, to prevent accidental over-reach during execution

Output to `docs/plans/YYYY-MM-DD-<action>.md` (or harness-specified plan path under plan mode). Do NOT execute. The plan is the artifact; execution is a separate user-approved step.

### Multi-file artifact directory

When the brainstorm's output is a multi-file directory (e.g., an `improve-loop` campaign, a plugin scaffold, a harness scaffold) rather than a single prose document:

- **Output path**: a new directory whose location follows existing repo convention (e.g., `campaigns/<name>/` for improve-loop campaigns; `<plugin-root>/<name>/` for plugin scaffolds). Confirm the convention by listing the parent directory before writing.
- **Required artifacts** (campaign example): one human-readable instruction spec (e.g., `program.md`) that captures the approved design verbatim, plus the executable scaffolding the design prescribes (harness script, runner script, fixtures). Generate the spec and the scaffolding in the same brainstorm — do not defer scaffolding to a follow-up.
- **Section-by-section approval applies to the spec only.** The executable scaffolding (harness, runner) is mechanical translation from the approved spec, not new design. Compound-move + auto-mode rules from Step 4 still apply to the spec; under auto mode, write the scaffolding immediately after the spec is approved without further per-file approval.
- **Do NOT commit the directory.** Leave it for user review like any other Step 5 artifact.
- **Smoke-validate** the executable scaffolding per `references/validate-executable-artifacts.md` before handoff.
