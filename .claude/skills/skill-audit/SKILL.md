# Skill Audit

Analyze a skill file against the work done in the current Claude Code session to determine whether the skill has issues, could be improved, or needs new features. Report only — never modify the target skill.

## Invocation

```
/skill-audit <path-to-skill-directory>
```

Example: `/skill-audit .claude/skills/improve-loop`

The argument is the skill directory path. The framework automatically resolves `SKILL.md` within it.

## Worktree Awareness

If working inside a worktree (e.g., `.claude/worktrees/<name>/`), ALL file paths — target skill reads, FOUNDATIONS reads, sibling skill reads for cross-skill checks, and follow-up edits — must use the worktree root as the base path. The default working directory is the main repo root; tool calls without an explicit worktree path will silently operate on main.

## Checklist

1. **Read the target skill** — perform the audit-phase Read with the Read tool. Sub-points:
   - **Which file to Read**: `<provided-skill-path>/SKILL.md` (e.g., for `/skill-audit .claude/skills/improve-loop`, Read `.claude/skills/improve-loop/SKILL.md`). Append `/SKILL.md` to the directory argument; do not Read the directory directly, which returns `EISDIR`.
   - **Why this Read is required even when framework-loaded**: the command framework may have already loaded SKILL.md content into context (visible above the `ARGUMENTS:` line), but the Edit tool tracks per-file Read state and will reject follow-up edits if the audit phase did not explicitly Read the file. Framework-loaded content does NOT satisfy the Edit prerequisite. Read tool responses of the form `File unchanged since last read` DO satisfy the prerequisite — the prior Read this session is what primes the state, regardless of whether subsequent Reads return full content or `unchanged`. The Read also validates that the file exists and is reachable; it is cheap (one tool call) compared to mid-implementation Edit rejection.
   - **What to extract**: parse the skill's name, description, and full content. If the file does not exist or is not a skill file, stop and report the error.
   - **Oversized SKILL.md**: if the file exceeds the Read tool's token limit, read it in sequential chunks covering the *whole file* — partial coverage leaves later regions unprimed for follow-up edits. Record section-heading landmarks (heading text + line numbers) while reading to support targeted re-reads during implementation. (This mirrors `skill-consolidate`'s Step 1.)
   - **Handling `references/` subdirectory**: if the skill directory contains a `references/` subdirectory, read all `.md` files in it — these contain the skill's extracted instructions and are the primary audit surface (and they too must be Read'd before any follow-up edits to them). The SKILL.md may be a thin entry point that delegates to references.
   - **Auditor-bias warning against skipping references**: the temptation to skip references that don't match obvious session evidence is a real auditor bias — references often contain the most recently-added rules, and the auditor's session-blind spots are a common source of missed findings. Reading all references at audit time (rather than deferring to follow-up implementation) is the safer default; deferral risks completing an audit that overlooks an entire reference's worth of structural issues.
2. **Read alignment documents** — Read `docs/FOUNDATIONS.md` unless its full text has appeared in a prior tool result (Read, or Agent tool result) this session. Partial excerpts from grep output and summaries embedded in CLAUDE.md do not substitute — re-read if you need the full principle text for cross-check. Do not rely on memory or training knowledge. `CLAUDE.md` is always available via system context injection and does not need explicit reading.
3. **Session reflection** — Review the current conversation context to identify:
   - Moments where the skill's instructions were unclear or ambiguous
   - Steps that were skipped, reordered, or worked around
   - Behaviors the skill didn't anticipate (edge cases, unexpected inputs)
   - Places where Claude had to improvise because the skill didn't provide guidance
   - Outcomes that diverged from what the skill intended
   - Steps that were not exercised this session (mark as "not exercised" — do not speculate about issues)

   **Direct vs. observational evidence**: when auditing a skill exercised earlier in this session, session evidence is direct (execution gaps, workarounds, improvisation). When auditing a skill being used for the first time in this session (including self-audit), evidence is observational — focus on: (a) instructions that could be read ambiguously, (b) missing guidance for edge cases visible in the skill text, (c) cross-skill consistency with workflow partners.

   **Self-audit special case**: If the target skill is `skill-audit` itself AND it was already exercised earlier in this session against another target (e.g., you audited some-other-skill, then turn the lens on skill-audit), evidence is direct from that prior execution — observational only when no prior exercise exists. For self-audit evidence-level determination (thin/moderate/rich), see the report template's "Evidence levels (self-audit of skill-audit specifically)" rubric — it differs from the normal-audit rubric. **Cross-execution triangulation under rich evidence**: when self-audit evidence is rich (≥2 prior auditor executions this session), explicitly look for gaps that surfaced in multiple prior audits — a skill rule that caused improvisation in two different target audits is stronger evidence than friction in one, and warrants more confidence in the severity tagging. Cite each prior execution in the finding's "What happened" section so the audit reader sees the recurrence pattern explicitly, not just a single anecdote.

   **Issue framing**: when identifying findings, note whether the gap is an over-constrained assumption (skill says X but should allow Y) or missing coverage (skill says nothing about Z) — this distinction guides whether the fix is relaxing a constraint vs adding new guidance.

   **Convergence scope**: Convergence (per the Summary template below) applies per-audited-skill, not per-invocation: self-auditing `skill-audit` after using it to audit another skill this session does NOT trigger convergence for `skill-audit` itself. Convergence requires `skill-audit` to have been previously audited AND updated this session — prior execution as an auditor does not count.
4. **Cross-check alignment** — For each finding from step 3, check whether the skill contradicts or fails to implement:
   - Principles from `docs/FOUNDATIONS.md` (reference by foundation number, e.g., "Foundation 7: Immutability")
   - Conventions from `CLAUDE.md` (reference by section name)
5. **Classify findings** — Categorize each finding into one of three buckets:
   - **Issue**: Something broken, misleading, or contradictory in the skill
   - **Improvement**: A refinement to existing behavior that would make the skill more effective
   - **Feature**: A new capability that aligns with the skill's stated intent but is currently missing

   When in doubt between Improvement and Feature: *Improvement* refines prose or output structure of an existing instruction. *Feature* adds a new capability path — something the skill couldn't produce before but now can. Structural templates or output-format refinements for existing behaviors are Improvements, not Features. *Example 1*: adding a new subsection (e.g., "Suggested follow-up") to an existing summary template is an *Improvement* — the summary already existed and is being structurally refined. *Example 2*: adding an entirely new output artifact the skill didn't previously produce is a *Feature*. *Example 3*: adding a new numbered step to the Process / Checklist — *Feature* if it enables a capability path the skill could not previously produce (e.g., a validation phase that gates output); *Improvement* if it refines an existing step's fidelity (e.g., splitting one step into two for clarity without changing the skill's effective reach).

   **Pattern-shaped findings — exhaustive enumeration required**:
   - **The rule**: When a finding (an Issue, or an Improvement whose fix substitutes a term with a now-superseded meaning) describes a *pattern-shaped defect* (a misused citation, wrong constant, deprecated terminology, stale TODO marker, etc.) that may recur across the target file, do not stop at the instances surfaced by session evidence alone. Run an exhaustive grep for the bad pattern across the target file (and the `references/` directory if present) BEFORE finalizing the finding's location list. Cite every instance found and note the enumeration as exhaustive in the report (e.g., "instances: lines 90, 93, 188, 250, 253 — exhaustive per `grep -nE 'FOUNDATIONS F[0-9]+'`").
   - **Execute the grep — don't cite it from visual inspection**: Use the Bash tool to actually run the grep before writing the citation. Citing a grep command without executing it (e.g., from visual inspection of the file you just Read) is a process violation; the implementation-phase sentinel grep will surface the gap, but at the cost of an extra round-trip and an inaccurate audit report. Without this, implementers may treat a non-exhaustive enumeration as authoritative and miss residual instances. **Anchor Bash paths**: skill-audit runs at end-of-session, after other skills may have changed the Bash working directory — anchor every `grep`/`ls`/file-touching Bash command in this skill (audit-time greps, the `references/` check, implementation-phase verification greps) with absolute paths, so they are robust to cwd drift. A bare relative path that silently resolves against the wrong directory produces a false-negative grep, not an error.
   - **Trigger heuristic**: if the finding's proposed fix involves *substituting* a term, name, identifier, or concept (not merely deleting a wrong reference), treat it as pattern-shaped — the substitution may need to apply at every occurrence of the OLD term where it carries the now-rejected meaning, not only where session evidence first surfaced it. Run the exhaustive grep at audit time, and the sentinel grep at implementation time, even when the OLD term remains valid in other contexts (in which case the enumeration distinguishes the "must-rename" instances from the "intentionally-retained" instances). **Concept/framing substitutions** (as opposed to a literal token) often have no single greppable signature — the rejected framing surfaces as several different phrasings, and a literal grep would either under-count or match dozens of legitimate uses. In that case the "exhaustive enumeration" is a documented manual section-scan, or a grep of the narrowest literal proxy *plus* a manual scan of adjacent prose for the concept's other surface forms; the report states which enumeration method was used, so "execute the grep — don't cite from inspection" is satisfied honestly rather than contradicted.
   - **Paired implementation-phase rule**: see Guardrails → Follow-up implementation → rule 5 for the sentinel-grep that closes the loop at edit time.
6. **Severity-tag each finding** — CRITICAL (skill produces wrong output or violates guardrails) / HIGH (common path confused or blocked) / MEDIUM (uncommon path confused or suboptimal outcome) / LOW (minor friction or edge-case gap). When a finding could fit two tiers, determine the base tier from the descriptions first — edge-case gap → LOW; uncommon-path suboptimal outcome → MEDIUM; common-path confused or blocked → HIGH; wrong output or guardrail violation → CRITICAL — then escalate by one tier if the skill gap caused improvisation, a misstep, or a user-visible workaround *this session*. The base-tier mapping above reads naturally for Issues (defects in the skill text). For Features (missing capabilities), base tier maps to *absence frequency*: edge-case absence → LOW; absence that occasionally forces auditor improvisation → MEDIUM; absence that frequently forces improvisation → HIGH; absence that prevents the skill from completing its core purpose → CRITICAL. The same one-tier escalation rule applies. *Escalation example*: a skill that omits an output enumeration but provides generic auto-mode fallback guidance still triggers escalation when the auditor had to invent specifics (a concrete next-step recommendation, a specific menu item, a path). *Second escalation example*: the audited skill prescribes `gh run view --log-failed` for log fetching; the auditor finds the command returns empty for cancelled runs and invents a `gh run view --log` fallback. The auditor invented a specific path the skill didn't name → escalate the finding's base tier (e.g., MEDIUM → HIGH) even though the workaround is small. Generic fallback reduces friction but does not eliminate improvisation. *Third escalation example* (Improvement tier): the auditor encounters an ambiguous wording rule (LOW base) — e.g., a "phase-boundaries table" prescription where strict tabular form vs. per-phase prose is unclear — and improvises a judgment call mid-session that turns out correct; the improvisation itself qualifies → escalate LOW → MEDIUM. The escalation rule applies symmetrically across Issues, Improvements, AND Features — Improvements are not exempt just because their fixes are wording-level. Skip escalation only when the skill text already names the specific path the auditor took. When the skill provides an explicit fallback (e.g., a verification step like Step 7 in a multi-step pipeline) that caught the issue without auditor improvisation, that's the skill working as designed — no escalation, even if the cost was an extra round-trip. The distinction matters: "auditor took the path the skill explicitly directed them to" is not improvisation, even if the path is corrective; "auditor invented a path the skill didn't anticipate" is.

   **Final escalation pass**: After tagging all findings with their base tier, do a single sweep across the finding list and ask, for each finding (Issues, Improvements, AND Features — the rule is universal, not Issue-only): did this gap cause improvisation, a misstep, or a user-visible workaround in *this session's execution*? If yes AND the base tier is below CRITICAL: escalate +1 tier. Record the escalation reason inline in the finding (e.g., `"base MEDIUM + escalated HIGH per session improvisation"`). The base-tier-first / escalation-second order is intentional — base tier maps the gap's intrinsic shape; escalation maps the session's actual cost. Conflating them on the first pass tends to under-tag features whose absence forced quiet improvisation.
7. **Present the report** — Output the structured report using the template below. **Pre-presentation check**: before presenting, confirm every finding whose proposed fix substitutes a term, name, or identifier has had its exhaustive grep executed (per Step 5) and its location list reflects all instances — a term-substituting finding presented without an executed enumeration is an incomplete audit report. This reinforces Step 5's "BEFORE finalizing the finding's location list" mandate at the natural checkpoint, rather than relying solely on the downstream sentinel grep.

Scale analysis depth to skill complexity. For small skills (<50 lines, <3 steps), the reflection can be a single paragraph. For large skills (>150 lines, >6 steps), each sub-item in step 3 deserves explicit consideration.

If 2+ findings target the same section heading or share a coherent theme (same failure mode, same user pain point), group them using either (a) a shared lead-in sentence before a flat list (e.g., "Three Step 2 Confidence-handling findings:") or (b) a thematic subsection heading per group (e.g., `### Step 2 Confidence-handling`) — both patterns preserve per-finding traceability. Each finding retains its own bullet, classification, and severity — the lead-in or subsection heading is presentation scaffolding only, not a merger. Single-finding groups stay flat — no lead-in sentence and no subsection heading — to avoid scaffolding inflation when a theme contains only one finding.

Grouping is scoped within each top-level section (Issues, Improvements, Features) — findings that share a section heading or theme but have different classifications stay in their respective top-level sections, since the classification taxonomy is the primary navigational structure of the report. Cross-classification thematic relationships, if useful for the reader, belong in the Summary's narrative (e.g., "Issue 1 and Improvement 4 are coupled: ...") rather than in a shared top-of-report subsection that would mix Issue and Improvement bullets under one heading.

## Report Template

Output this structure to the conversation (do not write to a file):

```markdown
# Skill Audit: <skill-name>

**Skill path**: <path>
**Session date**: YYYY-MM-DD
**Session summary**: <1-2 sentence description of what work was done with this skill. For observational / thin-evidence audits where the skill was not exercised as an auditor this session, describe the session context that motivated the audit instead — e.g., "Session used /reassess-spec and /spec-to-tickets; turned the lens on skill-audit before deploying it against a target.">
**Session evidence**: <rich / moderate / thin> (rubric: <normal audit / self-audit of skill-audit>)

*The two rubric blocks below are template-only guidance for the auditor — pick the rubric that matches the audit kind, apply it to determine the `<rich / moderate / thin>` value above, and do not include the bulleted rubric definitions in the rendered report.*

Evidence levels (normal audit):
- rich — skill executed 2+ times this session, or a single execution that exercised ≥3 of the skill's distinct branching paths (gate cycles, error-handling fallbacks, advisory branches, post-completion follow-ups).
- moderate — single full execution, or single execution with audit-driven edits applied to the target skill.
- thin — self-audit or first-time observation only.

Evidence levels (self-audit of skill-audit specifically):
- thin — skill-audit used zero or one time this session as auditor with no follow-up implementation.
- moderate — one prior execution as auditor plus follow-up implementation against some target (does NOT require the self-audit itself to have been implemented yet).
- rich — 2+ auditor executions this session, with or without follow-up implementation.

"Audit-driven edits" means edits to the *target skill* produced from this audit's own findings — not downstream use of the target skill's own outputs.
```

If analysis during classification disproves an initial impression, withdraw the finding inline with a brief explanation rather than omitting it silently — this documents the reasoning for completeness. If analysis only *partially* disproves the impression — the finding is still real but less severe than first tagged — downgrade the severity rather than withdrawing, and note the reasoning trail inline (e.g., "initially scoped HIGH; downgraded after analysis showed X covers the case"), the same documentation discipline as a full withdrawal.

## Guardrails

- **Report only** — Never modify the target skill file. Output the report to the conversation only.
- **No false positives** — If a step in the skill wasn't exercised during the session, note "not exercised this session" rather than speculating about potential issues.
- **FOUNDATIONS alignment is mandatory** — Any suggestion that would violate a principle in `docs/FOUNDATIONS.md` must be flagged and rejected, even if it would otherwise be an improvement.
- **Scope discipline** — Do not propose expanding the skill's scope beyond its stated intent. The audit evaluates the skill as written, not what it could become.
- **Evidence required** — Every Issue and Improvement must cite specific evidence — either *session evidence* (execution trace, improvisation, workaround, outcome divergence) OR, for observational / thin-evidence audits, *textual evidence*: the specific passage in the skill text that is ambiguous, self-contradictory, lacks edge-case coverage, or conflicts with a sibling skill. For Issues, cite what went wrong (session) or what the text says that misleads a reader (textual). For Improvements, cite where you hesitated, improvised, or made an arbitrary judgment call that the skill could have guided (session), or identify the passage whose wording could be sharpened (textual). Findings based purely on hypothetical scenarios — neither grounded in session execution nor tied to a specific passage in the skill — belong in Features, not Issues.
- **Follow-up implementation** — After the report is presented, the user may request implementation of specific suggestions. At that point, edit the target skill file directly — the "report only" guardrail applies only to the audit phase, not to user-directed follow-up.
  1. Ensure the target skill has been Read with the Read tool somewhere earlier in this session. The audit-phase Read in checklist Step 1 satisfies this — Step 1 unconditionally Reads SKILL.md (and any `references/*.md`) precisely so the Edit tool's per-file Read-state prerequisite is primed before any follow-up edits land — **unless a context compaction has occurred since that Read** (see point 7), which silently clears per-file Read-state. The Edit tool will reject edits without a prior Read; content visible from the command framework or earlier session context does NOT satisfy this check. A session interruption (point 7) is the one case requiring a fresh re-read before editing.
  2. Process edits top-to-bottom within the file to avoid offset drift. **Default to parallel**: batch all Edits in a single message whenever every Edit's `old_string` is unique across the file — `Edit` uses exact string matching, so offset drift is a non-issue when old_strings cannot collide. Sequential is the exception, used only when old_strings could collide (e.g., `replace_all` near a repeated phrase), when one edit's content becomes the next edit's anchor, or when insertions are adjacent enough that they might shift each other's anchor text.
  3. Combine or separate Edit calls using the following decision rules, evaluated in order (first match wins). Rule 2 and this rule are orthogonal axes that compose: Rule 2 decides message-batching (parallel = one message, the default), this rule decides Edit-call granularity (combine findings into one call vs. separate calls). "Separate Edits in top-to-bottom order" below means separate Edit *calls*, ordered top-to-bottom *within* Rule 2's single parallel message — not sequential messages. The common case is several separate Edit calls, all sent in one parallel message, listed top-to-bottom.
     - **Same skill location or coherent paragraph** (including in-place restructuring of a single bullet into nested sub-bullets, or splitting a flat paragraph into lead-in + child items) → combine into one Edit, even if findings are classified differently (e.g., an Issue fix and a Feature addition that form a unified section together).
     - **Different items within the same contiguous short numbered list (<10 items, one 1./2./3./4. sequence under one parent — not multiple discrete numbered code blocks within a parent section, each of which is its own list)** → combine into one Edit to avoid mid-list offset drift.
     - **Cross-section consistency edits for a single finding** (e.g., a numbered-rule change that also needs an update to a guardrail bullet cross-referencing the rule, or a menu item change that needs matching guidance in a "Not Exercised" section) → separate Edits in top-to-bottom order. Place the primary change edit first; the cross-reference edit follows so the referencing text matches the new text of the primary rule.
     - **Adjacent but independent bullets across separate logical sections** → separate Edits in top-to-bottom order (keeps each finding traceable).
     - **Default (no rule above applies)** → combine if adjacent or overlapping; separate if non-adjacent.
  4. When a finding requires edits across multiple files (e.g., cross-skill consistency fixes): if the cross-file edits are *independent* (no cross-file dependencies, no shared term substitution, each old_string unique within its file), Rule 2's default-to-parallel applies — batch all Edits in a single message regardless of file boundaries. If the cross-file edits are *dependent* (one file's edits affect the validity or content of another's — e.g., a renamed function called from another file, a constant changed in two places, a cross-skill consistency fix where canonical text in file A must match a reference to it in file B), process the *edit step* for each file independently — complete all edits in one file before starting edits in the next. The dependent branch is for *mutations* of existing shared text, where a stale-vs-fresh desync is possible; *adding* a new, consistently-spelled shared term across files (e.g., a new config key defined in one file and referenced in others) is order-independent — the *independent* branch applies. The *verification step* can be batched across files when the verification mechanism inherently covers all modified files atomically (e.g., a single grep command listing one unique phrase per Edit across the file set). After completing all files, verify cross-file references are consistent (e.g., step numbers referenced in SKILL.md match headings in reference files).
  5. After edits, verify by re-reading. Sub-rules:
     - **Default**: grep for a unique phrase from each edit to confirm it landed. When a single Edit's `new_string` introduces multiple discrete logical additions (e.g., two new sub-bullets in the same routing list, or two unrelated clauses inserted into the same paragraph), verify each addition with its own grep — one phrase per logical addition, not per Edit call. When the inserted content is also referenced from other locations within the same Edit (e.g., a new section heading also cited by N cross-references added in the same `new_string`), the expected hit count is `1 + N` (heading + cross-references). The grep's purpose is "did it land", not "does it appear exactly once" — a hit count matching the intentionally-added occurrences is a pass; pause to interpret only when the count exceeds the intentional-occurrence count (which would indicate the phrase already existed elsewhere and the edit may have collided).
     - **Override** for special cases: if edits are adjacent or overlapping, verify after each edit to catch offset drift; if the cumulative inserted or replaced content across all Edits exceeds ~50 lines AND any of {edits are adjacent or overlapping; edits restructure file-level layout (renamed top-level sections, wholesale region replacements); an Edit replaces a region containing another Edit's `new_string` content} also holds, do a full-file re-read instead of targeted greps. The Edit tool's exact-string semantics already prevent ripple effects when each `old_string` is unique and edits are non-adjacent — under those conditions targeted grep with one unique phrase per Edit is sufficient verification regardless of cumulative volume. Cumulative volume is the trigger for re-evaluating the override question — not the file-line distance between the first and last edit.
     - **Phrase selection**: pick unique phrases that avoid regex metacharacters (`*`, `?`, `[`, `(`, `|`, `^`, `$`, `.`, `\`) when possible — skill files contain markdown syntax (`**bold**`, inline code, bulleted lists) that collides with BRE/ERE interpretation and forces manual escaping. When a phrase must contain metacharacters, use `grep -F <phrase>` (fixed-string match) to avoid escaping errors. When verifying ≥5 phrases in a single shell loop (the typical batched-verification case), using `grep -F` for all phrases unconditionally is acceptable and often preferable — the per-phrase metacharacter audit cost exceeds the readability cost of the flag. Pick the phrase by copying a contiguous substring directly from the inserted `new_string`, not by paraphrasing — paraphrases easily drop markdown delimiters (`**`, backticks), whitespace runs, or hyphens that are present in the file. The simplest reliable phrase is a 4–8 word run of plain words from the middle of the inserted block. *Worked example*: if `new_string` contains `- **Scalar invariance**: multiplying a single consideration's weight by a constant preserves argmax...`, a BAD verification phrase is `"Scalar invariance: multiplying"` (drops `**` delimiters, leading-bold-marked term not in the file as plain text → 0 results) — a GOOD verification phrase is `"multiplying a single consideration's weight by a constant"` (plain-word run from the middle of the bullet, robust to markdown). When in doubt, scan past any leading bold-marked term to the descriptive prose that follows.
     - **Sentinel grep for pattern-shaped findings**: When the audit classified a finding as pattern-shaped (per Step 5 of the audit phase, "Pattern-shaped findings — exhaustive enumeration required"), after all per-edit landings are confirmed, run a single sentinel grep for the buggy pattern (e.g., `grep -nE '<bad-pattern>'`) across the modified file (and `references/` if present). Zero results required. If results remain, the audit's enumeration was incomplete — apply the missing edit(s) before declaring the implementation done. Per-edit phrase grep confirms each LISTED edit landed; the sentinel grep confirms no residual instances remain.
     - **Defensive sentinel grep**: even when no audit finding was classified pattern-shaped, run a quick `grep -nF '<old-term>'` for any term that an Edit explicitly *substitutes* with a new term whose meaning supersedes the old one (e.g., renaming a function, replacing a deprecated identifier, swapping a constant value, retiring a concept). Mid-phrase amendments where the original term retains its meaning (e.g., `or similar.` → `or similar unconditionally.`, or `or an existing skill` → `an existing skill, or a reports/ file ...`) do NOT trigger the sentinel — the original term's role is unchanged, only its scope is amended. All remaining results must be intentionally-retained occurrences (the auditor distinguishes "must-rename" from "intentionally-retained" instances). This is a common-sense backstop, not a process step — it costs one tool call and catches missed pattern-shape classifications.
  6. Watch for numbered list breakage — insertions commonly break numbering, create duplicate headings, or split contiguous lists. When inserting a new top-level section between numbered Steps, the core principle is to avoid renumbering existing numbered steps. Two insertion patterns satisfy this:
     - **Unnumbered descriptive section name** — e.g., adding "## Plan Mode Interaction" between "## Step 1.5" and "## Step 2". Appropriate when the target skill does not already use sub-numbered sections, or when the new section is a cross-cutting concern rather than a step in the primary flow.
     - **Sub-numbered insertion following existing convention** — e.g., adding "## Step 2.5: External Prior-Art Survey" between "## Step 2" and "## Step 3" in a skill that already uses "Step 1.5". Appropriate and often stylistically preferable when the target skill already establishes a sub-numbered pattern, and when the new section belongs in the primary step sequence.
     Pick whichever matches the target skill's established convention. Both patterns avoid renumbering and preserve intra-document references and external links.

     **Intra-Step sub-section insertion** is a different topology — when inserting between two unnumbered sub-sections within an existing Step, match the existing sub-section style (typically `**Bold lead-in**:` or `### Sub-heading`). No numbering concerns since unnumbered sub-sections aren't sequenced; intra-step insertions don't risk renumbering.

     **Intra-Step NUMBERED sub-step insertion** is a third topology — when adding a new numbered step to an existing intra-Step numbered list (e.g., inserting a new step 2 into a Step's existing 1-5 sequence). Renumbering is unavoidable here, unlike the top-level Step case. Accept the renumbering, and:
     (a) update internal cross-references within the same Step (e.g., `"step 4 below"` → `"step 5 below"`, `"step 2 above"` → `"step 3 above"`) as part of the same Edit's `new_string`, so the renumbering and the cross-reference fixup land atomically;
     (b) sentinel-grep for external references from other sections that point to specific intra-Step numbers (e.g., `grep -nF 'Step 7.7 step 4'`) to confirm no callers from outside the Step need updating — most intra-Step references stay scoped within the Step, but cross-Step citations do occur (Tracked-musings prohibition referencing After Campaign Completes step 4 is a real example);
     (c) prefer the numbered-step pattern when the new content is substantive enough to warrant numbered prominence (an auditor-mandated precondition, a gate, a discrete process step); use the unnumbered sub-section style instead when the new content is a clarifying note or aside that does not deserve a numbered position. The choice between numbered-renumber-and-update-refs and unnumbered-no-renumber is a content-weight judgment, not a process-friction judgment.
  7. If a session interruption occurred between audit report and implementation, re-read the target skill before editing. A "session interruption" includes a **context compaction / conversation summarization** — the most common cause in long multi-skill sessions — not only an external process touching the file. Two distinct reasons the re-read is required: (a) the file may have been modified by another process (content drift), and (b) a compaction silently clears the Edit tool's per-file Read-state priming, so edits are rejected with `File has not been read yet` even though the file is unchanged. Re-read every file you intend to edit, including each `references/*.md` — Read-state is tracked per file, and editing SKILL.md does not re-prime its references.
  8. If the system enforces plan mode, write a brief plan listing edits top-to-bottom, then execute after approval.
  9. The user may run a subsequent `/skill-audit` on the same skill after edits — this is a normal audit-edit-reaudit workflow.
  10. Under a "no clarifying questions" directive or auto-mode: if the user invokes "implement suggestions" against a mixed set (in-place fixes plus recommendations deferred to other skill invocations such as `/skill-extract-references` or `/skill-consolidate`), default to implementing the in-place subset and noting the deferred items in the summary. Don't ask for confirmation; the audit's own deferral note is sufficient signal.
- **Cross-skill consistency** — If the target skill is part of a multi-skill workflow, scan sibling skills for inconsistent file references, terminology, or shared constants. Report cross-skill inconsistencies as Issues. Sub-rules:
  1. **Defining sibling skills**: sibling skills are those in the same explicit workflow — triple patterns (e.g., `*-evaluate`, `*-plan`, `*-implement`), complementary pairs (e.g., audit/consolidate), or any skills where at least one explicitly names the other as a workflow partner. The naming need not be bidirectional: an upstream skill that produces the target's input (and names the target) is a sibling even if the target does not name it back.
  2. **Workflow chains**: if the target skill explicitly invokes or recommends a specific downstream skill by name (e.g., a Step 6 menu that says "invoke spec-to-tickets" or "invoke writing-plans", or a "Suggested next step" line naming the next command), those downstream skills are chain neighbors. For chain neighbors, read the downstream skill's input contract (typically its first numbered step or Invocation section — under ~30 lines) and confirm the target's output shape matches what the downstream expects.
  3. **Skipping the chain-neighbor read**: skip only if the output shape is a well-known convention (e.g., `specs/<NNN>-<name>.md`, `tickets/<PREFIX>-<NNN>.md`) already validated elsewhere in this session, or the chain neighbor's input is implicit session context with no positional artifact (typical of design-handoff skills like `writing-plans`).
  4. **Conditional follow-up offers**: offers like "optionally end the reply with `/schedule` if X" or "offer to draft the spec" are NOT chain neighbors for input-shape compatibility purposes — they offer the user a next action rather than handing off a structured artifact. Verify the chain neighbor only when the target skill produces an output that the downstream skill is expected to consume; soft conditional offers can be treated as standalone references that don't require the downstream input-contract check.
  5. **Unnamespaced downstream references**: if a downstream skill reference is unnamespaced (e.g., `writing-plans` rather than `superpowers:writing-plans`), resolve via the session's available-skills list to find the actual chain neighbor. **If the registry shows a namespaced form of the skill** (e.g., a `plugin:skill-name` variant exists): flag the bare reference as an Improvement on the target for explicit namespacing — this prevents resolution drift if multiple skills with similar names later coexist. **If the registry shows the skill is registered unnamespaced** (no namespaced variant exists): the bare reference IS canonical — record `no findings` for that reference, do not flag. When flagging is warranted, this is a term-substitution finding per Step 5's trigger heuristic — run the exhaustive enumeration at audit time and the sentinel grep at implementation time per Step 5's pattern-shaped findings rule, distinguishing must-rename from intentionally-retained occurrences (e.g., backticked descriptive uses of the skill name elsewhere in the target's prose are intentionally retained).
  6. **Meta-skills operating on other skills**: for the `skill-*` family, the downstream input is typically a skill directory path — treat this as a well-known convention and skip the downstream read unless the skill explicitly declares a non-standard argument shape. This is input-format compatibility only, not full bilateral consistency, since the relationship is unidirectional.
  7. **Standalone skills**: skills with no workflow partners do not require cross-skill checks — note "standalone skill, no cross-skill check needed" and move on.
  8. **Self-audit of skill-audit**: when auditing skill-audit itself, the cross-skill check applies to skill-consolidate (its complementary pair). Verify that skill-consolidate's description of its relationship to skill-audit (in its introduction) accurately reflects skill-audit's current behavior and scope. **If you find a mismatch** (e.g., skill-consolidate describes skill-audit's outputs in terms that no longer match the current Issue/Improvement/Feature taxonomy, or attributes capabilities to skill-audit that have since been removed), report it as an Issue under this cross-skill check, and the suggested fix should propose corrected wording for skill-consolidate's intro — not a change to skill-audit, since the audit target is skill-audit. **If no mismatch is found**, record the cross-skill-check result as `checked: skill-consolidate — no findings` per the report template's option 2 (Alignment Check section). Self-referential audit is valid but findings should focus on the skill's instructions, not its meta-properties.
  9. **Skills with `references/` subdirectories**: these were likely created by `skill-extract-references` — the reference files are the primary instruction surface, not SKILL.md. When checking such a sibling for cross-skill consistency, treat the sibling's `references/*.md` contents as part of its instruction surface — read the relevant reference doc(s), not just SKILL.md, before judging whether the sibling's behavior matches the target's documented expectations of it.
