# Bounded Local Refactor Fast Path

When ticket triage confirms a **bounded local refactor**, use this lean path unless reassessment later proves the boundary wider:

1. Read `docs/FOUNDATIONS.md`, the ticket, referenced specs/docs/Deps, and `AGENTS.md`.
2. Inspect repo state early and validate ticket-named files/functions/commands against the live codebase.
   - When the ticket changes a public helper contract, exported JSDoc, or another API-surface guarantee, do one quick export-surface sweep for sibling public wrappers, batch variants, or re-exported entrypoints that may inherit the same contract. Do not assume the draft ticket's seed list is the full public surface when the live package exports a wider helper family.
   - When adding a helper export, classify the export surface before proof: `same-file private`, `same-package internal source export`, `test-only/internal`, or `package public/barrel`. For same-package internal exports, a grep/importer/barrel check is usually enough; for package public/barrel exports, include downstream or emitted-surface proof and record the widened touch in the active ticket outcome.
   - For sibling-file extraction that turns same-file private helpers into same-package internal exports, use this compact checklist before final proof: classify each export as `same-package internal source export`; verify no package barrel/public export changed; run `rg` for old and new importers; prove with build/typecheck; and record the internal-export classification in the active ticket outcome when it affects touched-file or public-surface expectations.
   - When the ticket migrates an internal helper shape, especially `runtime -> runtime` to `state -> state`, grep for all direct writers of the same runtime or structural fields before coding or first proof. Update equivalent semantic insertion paths too, even when they bypass the named helper, so the new source-of-truth contract cannot be partially applied.
   - When the ticket replaces a custom loop/helper/probe with a canonical primitive, compare the old path's implicit defaults against the primitive's defaults before coding or first proof. Check trace retention, delta and snapshot work, hook/profiler behavior, runtime forking, kernel options, retained logs, and diagnostic shape; preserve the old lightweight behavior unless the ticket explicitly owns the broader behavior.
   - When the ticket adds a new test file, verify the correct suite family against nearby witnesses and the live lane manifest before coding, not just the literal draft path. In this repo, `unit`, `integration`, `determinism`, `performance`, `memory`, and `e2e` encode proof ownership, so a draft example path may point at the wrong family even when the filename stem is otherwise fine. Also inspect nearby test headers and the corpus marker rules for required `@test-class`, `@witness`, and `@profile-variant` adjacency before first proof; `test-class-markers` failures usually mean stale or missing file metadata, not a reason to weaken the guard.
   - For small test-only regression tickets that must exercise a production lifecycle or public entrypoint, classify setup controls before coding as `allowed deterministic preconditions` or `forbidden shortcut`. Allowed controls can include seed, deck/order selection, empty initial zones, or minimal authored fixture state when those only make the public path deterministic. Forbidden shortcuts include hand-writing the post-lifecycle state, visibility, trace, cache, or lookahead values that the ticket is supposed to prove. Add at least one intermediate assertion showing the production lifecycle/public seam actually ran before asserting the final invariant.
   - When a new test imports a helper or fixture from a different suite directory, classify the choice before coding as `reuse existing test helper`, `extract shared helper`, or `duplicate minimal fixture`. Prefer reuse when the helper is already a stable constructed fixture and no lane ownership is blurred; extract only when multiple suites need a neutral owner; duplicate only for a genuinely tiny fixture. Prove the target lane discovers and runs the new test.
   - For docs-plus-fixture tickets, classify whether the owned proof should reuse or extend an existing retained test instead of adding a new test file. If the ticket says no new tests but a live fixture/test witness already exists, prefer modifying that witness, record `no new test file; existing witness modified`, and prove the retained test still exercises the runtime/compiler seam rather than only importing the fixture.
   - When adding a canonical fixture helper or expanding a shared fixture file, check the target file size before editing. If the file is already near repo guidance or the ticket asks for a separable fixture owner, choose a sibling helper file; otherwise record why local extension is still bounded. Include a small source-size ledger in the active ticket outcome when file-size guidance affects placement.
   - When a new compiled test reads non-TypeScript sidecar artifacts such as JSON snapshots, markdown fixtures, corpus rows, or checked-in reports, verify before the first focused proof whether the package build copies those files into `dist`. If it does not, read the source-tree artifact through an existing fixture reader or a source-tree-relative path instead of assuming `dist/test/...` contains the sidecar.
   - When hand-authoring compiled fixture objects, inspect current compiler output, a nearby compiled fixture, or the authoritative type definitions before coding. Do not copy authored YAML/GameSpecDoc syntax into compiled runtime AST/catalog shapes. In this repo, for example, compiled `microturn.option.value` entries use structured intrinsic shapes, and optional properties should usually be omitted rather than set to `undefined` under `exactOptionalPropertyTypes`.
   - If build/typecheck failures are caused by hand-authored compiled fixture shape, classify them as fixture setup drift and fix them before counting the run as behavioral proof. Do not weaken assertions around the ticket-owned invariant to accommodate an invalid fixture.
   - When the ticket names a public API, alias, wrapper, or runtime entrypoint that the new test must exercise, write a quick witness map before coding: `ticket-promised witness API`, `actual assertion entrypoint`, `lower-level helpers used`, and `why this still proves the public contract`. Do not close on a lower-level helper assertion alone unless the ticket is corrected or the user authorizes that witness substitution.
3. Do a **command sanity pass** for ticket-named verification commands. You do not need the full `references/verification.md` load yet if the checks stay straightforward.
   - Prefer reading `package.json`, runner scripts, or lane manifests before probing package-manager commands with flags such as `--help`; do not assume script-level `--help` is non-executing.
   - Distinguish `the command is repo-valid` from `the command is runnable right now in the current artifact state`. If a lane depends on generated `dist/` output or another mutable build artifact, verify that prerequisite state explicitly before using the lane as the first acceptance-proof run.
   - In this repo, if a ticket names a stale focused engine proof command, prefer replacing it with the concrete runner shape the package actually supports: build first, then run `node --test dist/<focused-test>.js` from the package root, and record the correction in the active ticket outcome before final proof. From the repo root, use the equivalent `node --test packages/engine/dist/<focused-test>.js`, or use `pnpm -F @ludoforge/engine exec node --test dist/<focused-test>.js` when package-cwd semantics matter.
   - If a focused built-file `node --test dist/<focused-test>.js` run reports only a top-level `test failed` without nested assertion details, use the quick diagnostic fallback from `references/verification.md`: run the built module directly with `node dist/<focused-test>.js` from the package cwd when safe, then rerun the proper `node --test` lane after the fix before citing final proof.
   - In this repo, focused `node --test` lanes that exercise FITL, production `runGame`, `runGameSteps`, random-play helpers, or simulator-heavy production specs can still be slow and silent. Use an explicit `timeout` unless the exact lane is known-fast in the current session, and decide up front how a timeout will be classified or narrowed.
   - In this repo, when you need a truly focused runner proof lane, do not assume `pnpm -F @ludoforge/runner test -- <file>` narrows Vitest to that file. Prefer `pnpm -F @ludoforge/runner exec vitest run <file>` after confirming the package script/runner wiring, and record any substitution in the active ticket outcome before final proof.
   - When the ticket extends a helper with a diagnostic sibling, widened return shape, or other comparison-only contract, confirm the exact compared types from the canonical runtime surface before coding. Do not infer imported type paths or serialized-vs-runtime field shapes from draft prose when the live helper is meant to mirror an existing authoritative surface.
4. Load `references/implementation-general.md` only if the ticket widens beyond the simple local slice, exposes sibling/follow-up ownership drift, or otherwise needs the broader series guidance.
   - For bounded tickets inside an active series, use a compact sibling check before deciding no series edits are needed: dependency state resolved, residual work still owned by named siblings, no active contract/interface/command change leaks into siblings, and no spec ticket-list or dependency prose needs correction. Record `no sibling edits needed` in working notes when all checks are false.
   - When the active ticket outcome corrects a stale authoring noun, example shape, or contract term, run a narrow same-series prose sweep before final proof. Search the resolved spec plus active same-series sibling tickets for the corrected noun/example, update only same-contract drift, and record whether the sweep changed sibling/spec prose or found no sibling edits needed.
5. Load `references/draft-handling.md` for bounded local refactors only when draft status creates real boundary uncertainty:
   - the draft ticket/spec wording appears stale or contradictory
   - multiple sibling drafts may have overlapping ownership
   - the ticket needs durable draft-ticket correction beyond the normal working-notes checkpoint and closeout
   Otherwise, it is sufficient to record draft status in working notes and correct the active draft ticket during closeout.
6. Load the full `references/verification.md` only when verification planning becomes nontrivial because of shared outputs, multi-lane acceptance proof, migration fallout, or tooling ambiguity.
   - For straightforward bounded local refactors, prefer a cheap proof order: local package build or compile check first, then the narrowest focused test/proof lane for the owned change, then the package-level suite, then workspace-wide lanes last.
   - For docs-plus-fixture tickets, keep documentation examples adjacent to the existing relevant section, preserve Foundation-aligned generic wording, and avoid game-specific ids unless the ticket owns game documentation. Pair the docs proof with the smallest retained runtime/compiler witness that exercises the canonical fixture; a source export or type-only import is not enough when the ticket promises runtime fixture coverage.
   - When the owned change is documentation-only or contract-surface-only (for example JSDoc, assertion markers, helper boundary wording, or other intended no-behavior-change normalization), prefer `build` plus the smallest retained witness on the owned helper/runtime seam, then package `lint` and `typecheck`, before considering broader package/workspace `test` lanes. If the draft ticket names broader commands, rewrite that proof substitution into the active ticket before final proof.
   - For docs-only/manual-review tickets, make the manual proof concrete instead of treating it as a vague read-through. Identify the author-facing invariant, inspect the final prose/snippet against the live source/spec fact it describes, record any no-generated-fallout conclusion, and transcribe the exact review target in the active ticket outcome. Examples: a cookbook example includes the intended config flag, ref names, non-scored option caveat, and cost formula; a contract doc names the current public surface rather than a stale draft noun. When acceptance says an example "compiles", "is accepted", or otherwise matches a live contract, identify the canonical oracle before final proof: the schema source, type definition, compiler validator, generated artifact, or retained compile witness that owns acceptance for that authored shape. Record that oracle path or command in the outcome so the compile-clean claim is not just prose review. When the ticket requests a markdown preview and no GUI or markdown checker is part of the repo workflow, use a non-GUI source-review substitution only after inspecting the final section structure, fences, tables, links or anchors, and snippet boundaries; record the exact section reviewed and why the substitution satisfies the preview deliverable. If a markdown lint/check script exists, run it instead of relying on review alone.
   - When the owned implementation slice and verified fallout stay inside one package, prefer rewriting draft acceptance commands from workspace-wide `pnpm turbo ...` lanes to the narrowest package-local build/typecheck/lint/test lanes that fully prove that package-owned boundary only when the broad command is stale, unavailable, overbroad for the corrected owned invariant, or explicitly corrected in the active ticket. Do not replace a ticket-literal broad acceptance command merely because the diff is package-local; `AGENTS.md` Ticket Fidelity still requires every explicit command or deliverable to be run, classified, or truthfully corrected before terminal status.
   - If the ticket names only one focused proof file but the live diff changes additional owned seams that the named file does not exercise directly, add the smallest adjacent retained witness for each uncovered seam and record that proof-scope correction in the active ticket outcome before final proof.
   - When focused proof lanes, package suites, or workspace lanes consume freshly built `dist` output or another mutable package artifact, do not run them in parallel with the build step or with other commands that rebuild the same package. Finish the producing build first, then run the dependent proof lanes, package suite, and workspace lanes against that completed artifact set in sequence.
   - If an acceptance command cleans or rebuilds a shared output tree that another acceptance command reads from, treat that subset as order-sensitive even if the commands would otherwise be parallelizable. Shared `dist/` consumers are the common case: build first, then read-only proof lanes afterward.
   - For export-only or public-surface-only changes where a later accepted lane rebuilds `dist` after broad tests consumed an earlier build, use the narrowest final emitted-surface witness instead of reflexively rerunning the broad suite. Good final witnesses include a runtime import smoke against the emitted JS and/or a `.d.ts` check for the exported symbol. Record why the focused emitted-surface proof is sufficient and why behavioral broad lanes do not need rerun.
   - Even when commands do not rebuild shared artifacts, do not fan out multiple expensive acceptance lanes in parallel when they overlap the same large corpus. Prefer one broad lane or one focused lane at a time, especially in constrained environments such as WSL2 or other memory-limited sessions.
7. Still emit the full working-notes checkpoint before coding and still perform the final acceptance sweep before closeout.
   - For a small tracked engine test-only ticket, a compact checkpoint is enough when the full ledger would add noise: `status/spec/deps`, `ticket deliverables`, `suite/lane discovery`, `production lifecycle or public-seam witness map`, `generated fallout`, `proof lanes`, `terminal plan`, and grouped `not applicable` fields. Do not omit deliverables or proof-noun alignment just because the code change is only tests.
8. If the active ticket is an untracked or draft ticket that you expect to mark `COMPLETED`, `BLOCKED`, or otherwise durably rewrite at closeout, load `references/closeout-and-followup.md` before the final acceptance-proof pass so the ticket update lands before the last green run rather than invalidating it afterward.
   - For a tiny bounded draft Phase 0 slice whose owned work is only a Foundations/spec prose amendment, one adjacent constant/type/registry export, and one focused architectural test, use a compact bounded-draft closeout path. Still read `docs/FOUNDATIONS.md`, the active ticket/spec, `AGENTS.md`, and same-series siblings far enough to confirm deferral. Keep the pre-edit checkpoint complete but concise: draft/untracked status, exact deliverables, sibling deferral, generated fallout, command substitutions, source-size risk for the touched source file, and final proof lanes. Before final proof, prewrite the draft ticket outcome with touched-file scope, no-generated-fallout, sibling owners, source-size rationale, and exact proof plan. Then run final lanes serially with build before `dist` consumers, apply terminal status/proof transcription as the final narrow edit, run dependency integrity when status/deps changed, and finish with untracked-aware status plus untracked-file hygiene checks.
9. If live proof shows that a ticket-owned witness input (seed set, benchmark case list, pinned exemplar, historical repro id, etc.) is stale and the user approves a re-blessing path, keep the change narrow:
   - Prefer the smallest credible candidate search first: already-curated nearby seeds/cases, the previous witness family, or explicitly referenced historical candidates before broad brute-force probing.
   - Capture exact old-vs-new witness evidence during reassessment so the ticket correction can cite why the original witness drifted under the live code.
   - Update the active ticket's correction ledger and witness description before the final acceptance-proof pass so the last green run matches the re-blessed artifact.
   - If the ticket only asked for a representative sample (`roughly 4-6 seeds`, `small cross-game corpus`, or similar) and did not bind ownership to exact candidates, first do a cheap fresh-path probe of the candidate pool. Keep the final set small, representative, and passing on the owned invariant; exclude unrelated failing candidates and record the substitution explicitly in the ticket outcome.
   - When the stale witness is a recorded benchmark or performance ceiling rather than a semantic correctness example, first distinguish `stale recorded budget` from `live runtime regression`:
     1. compare the failing witness against nearby control cases or previously stable exemplars in the same harness
     2. check whether the owned implementation actually changed the measured hot path or only invalidated an old stored baseline
     3. rebaseline only when the evidence shows the harness expectations are stale while the live runtime contract remains correct
     4. once the previously failing witness now terminates honestly, classify the remaining gap explicitly as either `correctness/pathology still unresolved` or `budget now stale after the fix`; only rebaseline in the second case
   - Record that classification explicitly in working notes and the active ticket correction ledger so rebasing does not look like silently relaxing a performance guard.

## Baseline Snapshot Artifacts

For bounded tickets whose owned deliverable is a committed baseline snapshot or byte-identity fixture, use this compact pattern unless the ticket names a stronger generator:

1. Generate the candidate artifact from the same public fixture, harness, or runtime seam that the retained test will exercise; avoid hand-authoring repeated rows when the live seam can emit the canonical shape.
2. Keep the artifact normalized and deterministic, usually pretty JSON with stable key ordering from the runtime serializer or canonical helper already used by the test.
3. Run the focused test against the checked-in artifact, not just the generation probe, so missing sidecars, source-vs-`dist` path issues, and stale fixture reads fail before broad lanes.
4. Record the artifact provenance in the active ticket outcome: source fixture or seed, whether the baseline is historical, regenerated, or simulated from a current disabled/default path, and the command that proved the checked-in file.
5. If the checked-in artifact replaces an exact historical baseline or a ticket-named live simulator witness, treat that as a witness-deliverable correction and use the normal `1-3-1` boundary reset unless the ticket already allows that substitution.
