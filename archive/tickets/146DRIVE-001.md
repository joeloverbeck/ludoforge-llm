# 146DRIVE-001: Add `PreviewDriveResult` and `PreviewDriveOrigin` types

## Problem

Spec 146 introduces a new kernel-internal API `applyPreviewDriveGreedyChooseOne` for bounded greedy-chooseOne synthetic-completion drives. Its return type and origin context type need a stable home in `kernel/microturn/types.ts` so the implementation ticket (146DRIVE-002) and the test ticket (146DRIVE-003) can both reference them without circular drafting.

This ticket is the type-only seed: zero behavior, zero consumers, exists to unblock parallel authoring of implementation and tests.

## Architecture Check

1. **Cleaner than alternatives**: Co-locating the new types in `kernel/microturn/types.ts` follows the existing convention (all microturn-scoped types — `DecisionStackFrame`, `DecisionContext`, `TurnId`, etc. — live there). A separate `drive-types.ts` would fragment the namespace for two trivially small interfaces.
2. **Agnostic boundaries preserved**: Both types are pure structural definitions referencing only kernel-owned types (`GameState`, `SeatId`, `TurnId`). No game-specific data, no compiler interaction. Engine remains agnostic per Foundation 1.
3. **No backwards-compat shim**: Types are purely additive; no existing alias replaced or deprecated. Foundation 14 satisfied trivially.

## What to Change

### 1. Append two interface declarations to `packages/engine/src/kernel/microturn/types.ts`

```ts
export interface PreviewDriveResult {
  readonly state: GameState;
  readonly depth: number;
  readonly kind: 'completed' | 'stochastic' | 'depthCap' | 'failed';
  readonly failureReason?: string;
}

export interface PreviewDriveOrigin {
  readonly seatId: SeatId | '__chance' | '__kernel';
  readonly turnId: TurnId;
}
```

Place after the existing `DecisionStackFrame` / `DecisionContext` block; group with related microturn-scoped types. If `GameState` is not already imported in this file, add the import from `../types-core`.
