# 147AOTCON-001: Add compiled policy expression descriptors and equivalence scaffold

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Large
**Engine Changes**: Yes — compiler, kernel type/schema, and agent policy evaluator surfaces
**Deps**: `archive/tickets/146DRIVE-005.md`, `specs/147-aot-consideration-ast-compilation.md`

## Problem

146DRIVE-005 confirmed that the remaining post-146 FITL preview-pipeline hard-target miss is not a small missed `driveSyntheticCompletion` migration. Fresh V8 profile evidence on the production-profile hard-target corpus still shows `policy-evaluation-core:evaluateExpr` / `resolveRef` and preview-surface resolution dominating the agent side, while kernel query/hash work remains the generic downstream cost.

Spec 147 owns the next implementation slice: replace repeated interpretive consideration AST evaluation with compiled, typed descriptor trees that can be materialized into stable runtime closures without `eval` or game-specific branches.

## Assumption Reassessment (2026-04-26)

1. `packages/engine/src/agents/policy-evaluation-core.ts` still evaluates policy expressions through the interpretive `evaluateExpr` / `resolveRef` path.
2. `GameDef.agents` does not yet carry compiled policy evaluator descriptors, so no fixture/schema migration for Spec 147 has landed.
3. The 146DRIVE-005 profile assigns the next large owner to Spec 147, not to another narrow Spec 146 preview-drive patch.

## Architecture Check

1. The compiled evaluator must remain generic data: typed descriptor unions plus runtime factory functions, never generated JavaScript or `Function` / `eval`.
2. The compiler owns descriptor emission from GameSpecDoc policy IR; the kernel/agent runtime owns deterministic materialization and evaluation.
3. Per Foundation 14, once the compiled path is proven and migrated, repository-owned fixtures and consumers must move with it rather than preserving compatibility aliases indefinitely.

## What to Change

### 1. Add compiled descriptor types and schema coverage

Introduce the minimal `CompiledPolicyExpr` / `CompiledPolicyConsideration` descriptor surface needed to represent current consideration `weight`, `value`, `when`, and clamp/unknown behavior. Keep the descriptor union generic and serializable.

### 2. Add compiler lowering for existing policy expressions

Add a compiler pass that walks the existing policy library expressions and emits byte-stable descriptor trees alongside the interpreted IR. Do not change authored GameSpecDoc YAML.

### 3. Add runtime materialization scaffold

Add a runtime factory that can materialize descriptor trees into deterministic closures for the first supported expression families, with unsupported families failing loudly during development rather than silently falling back in production.

### 4. Add equivalence proof scaffold

Add tests that compare interpreted and compiled results for a focused corpus of existing FITL policy considerations before enabling the compiled path as the default.
