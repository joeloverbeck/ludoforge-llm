# 151ENVISO-001: Audit `node:*` importers and finalize Node-only/browser-safe classification

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — possible file renames in `packages/engine/src/sim/` and `packages/engine/src/agents/`; updates to barrel re-exports in `sim/index.ts`; updates to test imports for any renamed files
**Deps**: `specs/151-engine-package-environment-isolation.md`

## Problem

Spec 151 D3 enumerates six engine files with top-level `node:*` imports — two correctly named from PR #235 and four "at-risk" files that currently sit on browser-safe-by-default subpaths. Before the ESLint rule (003) and convention doc (002) can land, each at-risk file needs a per-file decision: rename to `*-node-loader.ts`, or rely on a directory-level ignore. The decision drives the ESLint ignores list and the doc's per-subpath classification, so the renames must execute before downstream tickets can finalize their content. Today's CI passes only because no other `node:*`-importing engine file is currently in the runner's transitive closure; this ticket establishes a stable post-rename file layout that the rule (003) will lint against.

3. `packages/engine/src/cnl/index.ts:23` re-exports `./load-gamespec-source.js`. CNL is Node-only by design and the runner does not import `@ludoforge/engine/cnl`, so the spec's plan to use a `cnl/**/*.ts` directory ignore (rather than rename) is consistent with this re-export remaining as-is.
4. Test consumers exist: `cnl/load-gamespec-source.ts` is imported by `packages/engine/test/unit/load-gamespec-source.test.ts` and `packages/engine/test/unit/staged-pipeline.test.ts`; `cnl/compile-observers.ts` is imported by `packages/engine/test/unit/cnl/compile-observers.test.ts` and `packages/engine/test/unit/cnl/compile-observers-zones.test.ts`. `policy-ir.ts` (sole importer: `packages/engine/src/cnl/compile-agents.ts:1`) and `sim/trace-writer.ts` have no test consumers.
5. Per-file decisions to confirm during implementation:
   - `sim/trace-writer.ts`: **recommended rename** to `trace-writer-node-loader.ts` (preserves explicit naming convention; the `sim/index.ts` re-export update is a one-line change). Alternative: add `packages/engine/src/sim/**/*.ts` to the rule's ignores (broader, covers any future sim-internal Node-only file).
   - `agents/policy-ir.ts`: **recommended rename** to `policy-ir-node-loader.ts` (single internal importer; the `agents/` subpath is browser-safe by default, so a directory ignore would over-broaden).
   - `cnl/load-gamespec-source.ts`, `cnl/compile-observers.ts`: **leave as-is**, covered by `cnl/**/*.ts` directory ignore (CNL is Node-only by design per spec line 99).

## Architecture Check

1. The audit decision space is bounded and small: each at-risk file has two viable options (rename vs. directory ignore), and the recommendations follow the spec's structural reasoning — rename when the file lives in a browser-safe-by-default subpath like `agents/`, directory-ignore when the file lives in a Node-only-by-design subpath like `cnl/`.
2. Foundation 1 (Engine Agnosticism): the renames make the file's host-environment status visible at the filename level, so future contributors classifying a new file see the convention by inspection rather than by failing CI.
3. Foundation 14 (No Backwards Compatibility): every rename migrates all callers in the same change — no aliasing, no transitional re-export shims. Renamed files keep their public symbols; only the import path changes for callers.
4. The ESLint rule (003) cannot land safely until the renames complete or the directory ignores are decided, because any browser-safe file with a `node:*` import would error against the new rule. This ticket establishes the post-rename codebase state that 003 lints against — the post-001 file layout is itself the decision record consumed by 002 and 003.
