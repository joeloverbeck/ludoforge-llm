# 151ENVISO-003: Add ESLint `no-restricted-imports` rule blocking `node:*` in browser-safe engine files

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — `eslint.config.js` only
**Deps**: `archive/tickets/151ENVISO-001.md`

## Problem

Spec 151 D2 calls for an ESLint rule that forbids top-level `node:*` imports in browser-safe engine source files, with exemptions for `*-node-loader.ts` files and Node-only-by-design directories. Without this rule, future contributors can re-introduce the failure shape PR #235 fixed; CI catches the violation only at the runner build step (downstream from the offending file), so the diagnostic loop is slow and confusing — the lint rule turns a multi-layer CI failure into an in-editor diagnostic at the offending line.

## Assumption Reassessment (2026-05-05)

1. `eslint.config.js` (root) already uses `no-restricted-imports` with per-glob `files` scopes — verified during spec 151 reassessment. The cited examples match the spec's claim exactly: lines 12–27 block `*contract*` imports under `cnl/**/*.ts`; lines 28–44 block `eval-error` imports under `effects-*.ts`. The new block follows the same pattern.
2. After 151ENVISO-001 lands, every browser-safe engine file is free of `node:*` imports — meaning this rule can land without producing immediate violations.
3. The ignore list depends on 001's decisions. At minimum it includes `packages/engine/src/**/*-node-loader.ts` and `packages/engine/src/cnl/**/*.ts`. Additional ignores (e.g., `packages/engine/src/sim/**/*.ts` if 001 chose ignore-over-rename for `trace-writer.ts`) are determined by the post-001 codebase state.
4. The known browser-safe file recommended for the deliberate-edit verification (AC #2) is `packages/engine/src/agents/policy-eval.ts` — verified to have no `node:*` imports in its current state.

## Architecture Check

1. Lint-time enforcement (vs. CI-build-time) catches violations at the offending file in editor diagnostics, before commit. Authors fix the issue locally rather than three layers downstream.
2. Foundation 1 (Engine Agnosticism): the rule operationalizes the engine-environment contract — engine code does not assume a host that ships `node:*` modules unless the file is explicitly marked Node-only.
3. Foundation 5 (One Rules Protocol, Many Clients): the rule preserves the engine's single-runtime-contract surface for browser hosts; only the bootstrap layer (Node-loader files, CLI scripts, tests) carries Node-specific code.
4. Foundation 14 (No Backwards Compatibility): no shims, no aliasing — the rule applies uniformly from the day it lands; pre-existing violations are fixed in 001 before this ticket activates the rule.

## What to Change

### 1. Add `no-restricted-imports` block to `eslint.config.js`

Insert after the existing `no-restricted-imports` blocks (currently at lines 12–44). Shape:

```js
{
  files: ["packages/engine/src/**/*.ts"],
  ignores: [
    "packages/engine/src/**/*-node-loader.ts",
    "packages/engine/src/cnl/**/*.ts",
    // Add additional Node-only directory ignores here, matching the
    // post-001 codebase state (e.g., "packages/engine/src/sim/**/*.ts"
    // if 001 chose ignore-over-rename for sim/trace-writer.ts).
  ],
  rules: {
    "no-restricted-imports": ["error", {
      patterns: [{
        group: ["node:*"],
        message:
          "Engine browser-safe modules must not import node:* APIs. " +
          "Move file IO / Node bootstrap to a sibling <name>-node-loader.ts " +
          "and have callers import that file directly. " +
          "See docs/engine-environment-isolation.md.",
      }],
    }],
  },
},
```

The exact `ignores` list is determined by the post-001 codebase state. The implementer reads the post-001 file layout, finalizes the ignore list, and confirms it matches the doc (002, if landed by then).
