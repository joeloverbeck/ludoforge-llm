# 151ENVISO-004: Smoke test for `node:*` leaks in runner build output (optional)

**Status**: COMPLETED
**Priority**: LOW
**Effort**: Small
**Engine Changes**: None — adds a CI-time smoke test only
**Deps**: `archive/tickets/151ENVISO-003.md`

## Problem

Spec 151 D4 proposes an optional belt-and-suspenders CI check that greps the runner's vite-build output (`packages/runner/dist/assets/*.js`) for any literal `node:fs`, `node:path`, etc. references or Vite browser-external stub artifacts. The static ESLint rule (003) catches top-level `node:*` imports but cannot catch dynamic patterns like `await import('node:fs')` or rollup configuration drift that re-introduces externalized references. A post-build grep complements the lint rule by catching anything that slips through.

Spec line 152 marks this work optional — "Defer unless the audit finds existing dynamic imports" — and the audit (001) verified no current dynamic `node:*` imports exist in engine source. This ticket is therefore at LOW priority with an explicit descope path.

## Assumption Reassessment (2026-05-05)

1. No dynamic `await import('node:*')` patterns exist in `packages/engine/src/` as of spec 151 reassessment — verified by recursive grep. The spec explicitly defers this ticket unless dynamic imports surface; landing it is precautionary against future drift, not closing an existing gap.
2. The runner build emits to `packages/runner/dist/assets/*.js` (Vite default output layout). Confirm the actual output path during implementation by inspecting the post-build artifact tree.
3. The runner has a `build` script in its `package.json`. CI invokes `pnpm turbo build`, which runs the runner build in topological order after the engine.

## Architecture Check

1. The smoke test catches a residual class of leaks — dynamic imports, ESLint-rule bypass, rollup configuration drift — that the static lint rule (003) cannot. Together they form defense in depth.
2. Foundation 16 (Testing as Proof): proves the architectural invariant ("no `node:*` references in the runner browser bundle") via automated check rather than assumption.
3. The check is grep-based and side-effect-free — no impact on engine, runner, or CI configuration beyond a single new step or script.
4. Marked LOW priority + descope path because the static rule (003) already provides primary enforcement; this is value-add only.

## What to Change

### 1. Add a runner build smoke-test script

Either (a) a new script under `packages/runner/scripts/` (e.g., `verify-no-node-imports.mjs`) invoked after build, or (b) a CI workflow step. Implementer picks the lighter-weight option after reading `.github/workflows/` to see existing patterns.

The script:
- Greps `packages/runner/dist/assets/*.js` (or whatever Vite outputs in this repo) for `node:fs`, `node:path`, `node:url`, `node:crypto`, `node:fs/promises`, etc.
- Also fails on Vite `__vite-browser-external` stub assets or references, because live Vite output for a dynamic `await import('node:fs')` does not preserve the original `node:fs` literal in the emitted JavaScript chunk.
- Exits non-zero (with offending file + match) if any pattern is found.

### 2. Wire into CI

Either add a step to the existing CI workflow that runs the runner build, OR add a `verify:browser-bundle` script to `packages/runner/package.json` and call it from `pnpm turbo verify` (or equivalent). Pick whichever path matches the existing CI pattern with the smallest diff.

### 3. Verify the check fires

Manually inject `await import('node:fs').then(...)` at the top of a runner source file. Run the smoke test. Confirm it fails with a clear message identifying the offending output file and pattern. Revert the test edit before commit; document the verification in the PR description.
