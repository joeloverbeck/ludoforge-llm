# DECINSARC-006: Remove legacy decision-param aliasing from engine test helpers

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes; one generic runtime binding fix was required in addition to test helper and test-suite cleanup
**Deps**: DECINSARC-001, DECINSARC-002, DECINSARC-003, DECINSARC-004, DECINSARC-005

## Problem

`packages/engine/test/helpers/decision-param-helpers.ts` still preserves pre-`DecisionKey` compatibility behavior by accepting alias-like move params and helper hints (`$bind#N`, macro alias names, and synthetic `#1` variants of canonical keys). That makes tests pass even when they are no longer expressing canonical runtime input. The runtime architecture is already centered on `DecisionKey`; the remaining drift is in helper behavior and the tests that depend on it.

## Assumption Reassessment (2026-03-13)

1. `decision-occurrence.ts` and `decision-id.ts` are already gone from `packages/engine/src/kernel/`; this ticket is not the main migration.
2. `ChoicePendingRequest` already uses `decisionKey`; the old occurrence fields are no longer present in engine or runner tests.
3. `decision-param-helpers.ts` already imports `parseDecisionKey`, but it still implements compatibility lookup (`decisionParamLookupKeys`) for alias-like params instead of requiring canonical keys.
4. The main remaining dependency is test coverage that still relies on helper-side alias acceptance:
   - unit coverage in `packages/engine/test/unit/decision-param-helpers.test.ts`
   - a small set of integration tests that pass alias-like params directly (confirmed examples include `fitl-cap-caps.test.ts` and `fitl-cap-cobras.test.ts`)
   - many FITL integration tests still use override predicates that match by loose `decisionKey.includes(...)`/`name` patterns; these are not old occurrence fields, but they are less explicit than canonical-key matching
5. Canonical `DecisionKey` formatting is already codified in `decision-scope.ts` and covered by `packages/engine/test/unit/kernel/decision-scope.test.ts`; this ticket should align helper/test inputs to that codec rather than reinterpret it.

## Architecture Check

1. A strict canonical-key helper is better than the current compatibility layer because it keeps tests honest: if a test provides non-canonical decision params, it should fail instead of being silently normalized by helper heuristics.
2. This is architecturally better than the current state. The runtime has already chosen `DecisionKey` as the sole identity; allowing test-only aliasing reintroduces a second, implicit interface and increases long-term drift risk.
3. The clean endpoint is:
   - helper lookup only by canonical `request.decisionKey`
   - tests asserting canonical serialized keys where needed
   - override predicates using explicit decision semantics, preferably canonical keys or parsed-key structure when the exact key matters
4. Ideal follow-up architecture, if the override predicates keep proliferating: add a tiny test-only matcher utility around `parseDecisionKey()` so integration tests stop using brittle `decisionKey.includes(...)` substring checks. That is worth noting, but only implement it here if the touched tests need it.

## What to Change

### 1. Tighten `packages/engine/test/helpers/decision-param-helpers.ts`

- Remove compatibility lookup that accepts alias-style keys (`$bind#N`, macro aliases, synthetic `#1` suffixes for canonical keys).
- Keep helper behavior centered on canonical `request.decisionKey` lookup plus explicit override rules.
- Preserve existing stochastic probing / compound-move handling only where it is still needed for test ergonomics.

### 2. Update tests that still rely on helper-side aliasing

- Rewrite `packages/engine/test/unit/decision-param-helpers.test.ts` so it asserts strict canonical-key behavior instead of legacy alias compatibility.
- Update any integration/e2e tests that pass alias-like params into `applyMoveWithResolvedDecisionIds()` so they provide the canonical decision keys expected by the runtime.

### 3. Strengthen regression coverage

- Add or adjust tests proving:
  - canonical `DecisionKey` params still resolve correctly
  - alias-style helper inputs now fail fast instead of being silently accepted
  - repeated/nested decisions remain addressable when callers use canonical keys generated by the runtime
  - stochastic/compound helper behavior still works after the simplification
- Property-style additions are optional here; only add them if they materially strengthen the canonical-key invariant without introducing a large new test surface.
