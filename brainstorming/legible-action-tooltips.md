# Legible Action Tooltips

We've implemented a system to generate action tooltips that are shown on hover when the user hovers the cursor over actions such as Train, Sweep, etc. The technical report is in reports/action-tooltip-natural-language-spec.md

The issue is that we wanted the tooltips to look more like game rules generated from the GameSpecDoc files, perfectly legible in English, but they are basically like reading code right now.

I asked ChatGPT 5.4, supposedly the most intelligent model, to figure out if there were ways of improving this. It said:

## ChatGPT's response

Yes. But it is not a solved problem in the sense of “feed the raw AST to an LLM and get good rule text back.” The solved part is the verbalization architecture: formal rules are first turned into a higher-level message plan, and only then realized as controlled natural language. Your current system skips that middle layer. Your report shows that tooltips are generated by rendering ActionDef / ActionPipelineDef ASTs, while your actual FITL rules already have more meaningful structure: stage-based pipelines and reusable macros such as select-spaces, resolve-per-space, sweep-loc-hop, and sweep-activation. That means you are verbalizing the execution tree instead of the rule structure. 

The closest research family is ontology/rule verbalization and data-to-text generation. The standard model there is three-tiered: atomic identifiers map to lexical entries, individual formal statements map to sentences, and groups of related statements map to paragraphs or sections. Controlled natural language is also materially easier for users than raw formal notation, and systems like NaturalOWL get readable end-user text by separating document planning, micro-planning, and surface realization rather than by pretty-printing formal code.

That literature is also blunt about what actually improves readability. Tokenizing identifiers helps a little, labels help more, but real sentence plans, ordering resources, aggregation, and referring-expression handling matter a lot. NaturalOWL reports that tokenized identifiers are inferior to proper labels and sentence plans, and that removing sentence plans or ordering information sharply hurts fluency, clarity, and text structure. Work on ontology lexica makes the same point from another angle: you need explicit lexical resources for context-specific wording and sometimes audience-specific wording.

There is also board-game-specific precedent. Ludii’s automatic manual generator translates a high-level game description language into English and explicitly says that direct translation from a more verbose, lower-level language like GDL is difficult, while their higher-level approach works effectively and should transfer to other high-level game description languages. AtDELFI, a game tutorial generator, uses a graph of rules/mechanics and found that this works for simpler games, but more complex games need higher-level understanding than isolated mechanics. Your GameSpecDoc is much closer to Ludii’s “tractable” side than to raw GDL, but only if you verbalize the high-level stages/macros instead of the raw expanded AST.

So the right design is this:

## 1. Add a semantic compression layer between AST and English

Do not generate tooltips from raw EffectAST / ConditionAST nodes. Generate them from a new intermediate representation — call it TooltipIR, RuleCard, or RuleMessageGraph.

For FITL, compile per pipeline profile, not per bare action id. train-us-profile and train-arvn-profile are different rules, so they need different static tooltip plans. For simpler games like Texas Hold’em, compile directly from ActionDef.

At this layer, normalize low-level execution idioms into player-facing rule messages:

- chooseN over mapSpaces → selection step

- forEach over selected spaces → per-space resolution

- moveToken from available-* → place

- moveToken from adjacent zones → move

- setTokenProp activity=active on underground guerrillas → activate

- negative addVar on a resource → pay / spend / lose

- shiftMarker supportOpposition → shift support/opposition

- if globalMarkerState cap_* ... → modifier

- let, bindValue, concat, telemetry counters, temporary trackers → usually hidden

That last point matters a lot. Your AST contains real execution scaffolding — temporary bindings, telemetry like adviseCount / airStrikeCount, and temporary flags like m48PatrolMoved — that should almost never appear in a player tooltip. That is a content selection problem, not a surface wording problem.

A good heuristic set for your existing pipelines is:

- stages named select-* → “Where / Choose”

- resolve-* or resolve-per-* → main action steps

- sub-action, optional-*, trail-improvement → optional follow-up

- cap-* stages or branches on global markers / leaders → modifiers

- *-telemetry → suppress from tooltip

That gets you much closer to a rulebook structure with very little extra machinery.

## 2. Split static rule text from dynamic legality text

Right now you are trying to make one tooltip do two jobs: explain the rule and explain whether it is currently legal. Those should be separate outputs from the same IR.

Use two artifacts:

- RuleCard: static explanation for an action/profile

- RuleState: current availability, active modifiers, current numeric instantiations, and blockers

The rendered tooltip should look like this:

- What this action does

- Modifiers currently active

- Why it is unavailable now or Why it is available now

- Debug view (collapsed raw AST / trace)

For the legality view, do not dump the whole condition tree in English. Evaluate the condition AST into a satisfaction DAG and extract a minimal blocker set. For an unsatisfied and, collect the unsatisfied children. For an unsatisfied or, show the smallest unsatisfied alternative set, not the entire disjunction. For not, render the violated positive condition. This is much closer to how accessible explanation systems verbalize proof trees: they introduce a few intelligible intermediate statements instead of exposing the whole derivation.

## 3. Put a small authorable lexicon into GameSpecDoc

Your report is directionally right about identifier humanization, but simple camelCase splitting is nowhere near enough. Research and prior systems both show that labels help, but sentence plans and lexical metadata matter more. NaturalOWL can tokenize CamelCase and use rdfs:label, but it still needs proper sentence plans because labels alone do not supply enough grammar or aggregation information. Domain experts customizing lexical resources was also central in robust KB verbalizers.

So add a small verbalization: block to GameSpecDoc. Something like this:

verbalization:
  labels:
    arvnResources: "ARVN Resources"
    supportOpposition: "Support/Opposition"
    terrorCount: "Terror"
    guerrilla:
      singular: "guerrilla"
      plural: "guerrillas"

  stages:
    select-spaces: "Choose target spaces"
    resolve-per-space: "Resolve each selected space"
    sub-action: "Optional sub-action"

  macros:
    sweep-loc-hop:
      class: movement
      summary: "Troops may also move along clear Lines of Communication into the target space."
    sweep-activation:
      class: activation
      summary: "Activate underground guerrillas based on sweep strength in the space."
    place-from-available-or-map:
      class: placement
      summary: "Place pieces from Available, or redeploy them from the map if none are available."

  sentencePlans:
    shiftMarker:supportOpposition:
      +1: "Shift {space} 1 level toward Active Support."
      -1: "Shift {space} 1 level toward Active Opposition."

The crucial bit is that macros should not just have a string summary. They should have a semantic class plus slots, so your planner knows whether a macro is a placement rule, a movement rule, an activation rule, a modifier, or hidden bookkeeping.

Use automatic humanization only as fallback. Anything user-facing should have an explicit label or sentence plan. Natural-language names, sentence plans, and section ordering are exactly the resources that prior verbalizers found most important.

## 4. Realize in controlled rule English, then optionally polish

The target output should be controlled rule English, not freeform prose. That means:

- stable terminology

- stable sentence shapes

- stable ordering

- no decorative synonym churn

- no hidden inference

For English, a deterministic realizer with hand-built templates is probably enough. If you want better morphology and agreement, use a SimpleNLG-class surface realizer. If you want future multilingual support, keep the message plan language-neutral and map it through a GF-style abstract syntax / resource grammar layer. That is exactly the route taken in multilingual controlled-language work, and recent logic/contract verbalizers explicitly use literal AST-to-GF translation followed by tree transformations for more natural output.

Use an LLM only after the content has already been fixed by the deterministic planner. Recent data-to-text work is clear on the trade-off: end-to-end neural generation improves fluency, but semantic fidelity and hallucination/omission are still real problems; recent hybrid systems explicitly use symbolic structuring to constrain the LLM before realization. So the rule should be:

deterministic content plan first, optional LLM paraphrase second, automatic verification last.

A safe policy is:

- runtime: deterministic only

- build-time: LLM may propose macro summaries or alternative phrasings

- shipping text: only accepted if it round-trips back to the same TooltipIR

## 5. Use progressive disclosure, but from the same traceable source

The tooltip should not be a wall of text. The board-game manual and ontology verbalization literature both use staged disclosure: show the main facts first, then more detail on request. NaturalOWL explicitly supports showing more facts when the user asks for more information.

A good tooltip layout is:

- 1-line synopsis

- 3–5 main steps

- modifiers (capabilities, leaders, momentum, special windows)

- available now / blocked now

- raw AST / trace (collapsed)

Keep trace pointers from every sentence back to the originating AST nodes and macro ids. That lets you preserve your current annotation system, lets power users inspect the exact source, and gives you a clean debugging path.

## 6. Evaluate like a verbalizer, not like a summarizer

Do not trust BLEU-style text metrics here. Recent logic-to-text work points out that automatic naturalness/readability metrics are often inconsistent with human judgment.

Use human evaluation on three tasks:

1. Faithfulness: Does the tooltip omit or add any rule content?

2. Scannability: Can a player answer “Where?”, “Cost?”, “What happens?”, “What modifies this?” quickly?

3. Actionability: In a scenario card, do players pick the correct action more often with the new tooltip than with the old one?

Also add compiler tests:

- every TooltipIR slot must be realized or explicitly suppressed

- every generated sentence must retain trace ids

- if LLM polish is enabled, the extracted message set must equal the pre-polish message set

With this design, a FITL tooltip becomes something like:

Sweep (US)
Where: Provinces or Cities outside North Vietnam. Limited Op: 1 space.

1. Choose target spaces.

2. For each selected space, move adjacent US Troops in. Troops may also enter via clear Lines of Communication.

3. Activate underground guerrillas there up to sweep strength; jungle halves the limit.
Modifiers: Cobras (unshaded) removes 1 extra enemy in up to 2 spaces. Booby Traps (shaded) may kill 1 US Troop per selected space on 1–3.

And the same machinery can verbalize a card-game action cleanly:

Raise
Available if you still have chips and your betting action has been reopened.
Choose a total bet at least the current bet plus the last full raise, up to your all-in amount. Pay the difference between that total and your current street bet into the pot. Your street bet and total bet increase, the current bet and last raise size update, and other eligible players are reopened. If this leaves you with 0 chips, you become all-in. 

That is the real test of generality: the same planner can handle FITL and Hold’em if it operates on domain-neutral semantic messages like select, move, place, pay, shift, reveal, choose, modifier, blocker. It will not work well if it operates directly on raw if / forEach / let / moveToken trees.

## Direct answers to the hard choices

- Template vs LLM-only: deterministic core, optional LLM polish. Not negotiable.

- Depth limit: do not use raw AST depth. Use a rhetorical budget: main step + one sublevel, then collapse.

- Macro descriptions: manual by default, autogenerated fallback, LLM suggestions offline only.

- Identifier humanization: fallback only. Real tooltips need explicit labels plus sentence plans.

- Conditional branches: choices stay inline under the relevant step; capability/leader branches move to “Modifiers”.

- Cross-game generality: yes, with a neutral TooltipIR; no, with raw AST templates alone.

- Live annotation: separate static rule text from current blockers/satisfied conditions.

- Localization: very doable if you keep an abstract message layer; painful if you freeze English too early.

- Power-user mode: absolutely keep a raw AST toggle.

- Bottom line: build a verbalizer, not a prettier pretty-printer. The winning stack is:

GameSpecDoc -> semantic TooltipIR -> content planner -> controlled rule English -> optional verified polish

The next engineering move that actually matters is defining the TooltipIR and the first 20–30 normalization rules. That is where most of the leverage is.
