# 145PREVCOMP-005: Trace and diagnostics for driver and gate

**Status**: PENDING
**Priority**: MEDIUM
**Effort**: Medium
**Engine Changes**: Yes — `agents/policy-diagnostics.ts`; data update in `campaigns/lessons-global.jsonl`
**Deps**: `archive/tickets/145PREVCOMP-001.md`, `tickets/145PREVCOMP-002.md`

## Problem

Per Spec 145 §I5 and reassess-spec Improvement #8: extend verbose policy traces so operators can see (a) how deep the driver went per candidate, (b) which completion policy ran, (c) the new `'depthCap'` and `'gated'` reason codes, (d) how many candidates the top-K gate suppressed at each microturn (`previewGatedCount`), and optionally (e) when a sampled gated candidate would have outscored the chosen candidate (`previewGatedTopFlipDetected`) — the over-suppression observability signal.

Without `previewGatedCount`, the "preview.topK is tunable" mitigation noted in Spec 145 §Risks is not actionable: tuners cannot see when the gate is biting. Without the optional flip-detection signal, the only feedback loop on top-K under-coverage is downstream metric regressions, which are slow and indirect.

This ticket also appends an entry to `campaigns/lessons-global.jsonl` describing the post-145 expected `previewOutcome: 'ready'` rate as a smoke signal for future campaigns.

## Assumption Reassessment (2026-04-25)

1. `policy-diagnostics.ts` lives at `packages/engine/src/agents/policy-diagnostics.ts`. It currently emits `phase1Score`, `phase1ActionRanking`, and other per-candidate fields (verified at `policy-diagnostics.ts:126-128`).
2. Trace schema in `kernel/schemas-core.ts` includes per-candidate metadata fields (`phase1Score?`, `phase1ActionRanking?` at lines 1756-1758). New fields (`previewDriveDepth`, `previewCompletionPolicy`, `previewGatedCount`, `previewGatedTopFlipDetected`) extend this metadata block.
3. `previewGatedCount` is per-microturn (count of candidates suppressed by top-K gate), not per-candidate. Schema update needs to attach it at the microturn-trace level, not the candidate level.
4. `previewGatedTopFlipDetected` is optional and sampled — emitted only when the underlying preview-runtime cache already holds the gated candidate's outcome (i.e., a subsequent pass within the same evaluation has materialized the outcome). The optional/sampled framing avoids paying the gate's cost-savings back at observation time.
5. `campaigns/lessons-global.jsonl` exists; entry format is JSONL with `lesson` and metadata fields per existing entries.

## Architecture Check

1. **F#9 (Replay, Telemetry, Auditability)** — new fields are structured and deterministic; they do not introduce new event types, only enrich existing per-candidate and per-microturn metadata.
2. **F#11 (Immutability)** — diagnostic emission is read-only over the runtime state already produced by the driver and gate.
3. **F#12 (Compiler-Kernel Validation Boundary)** — schema additions in `kernel/schemas-core.ts` are validated alongside existing per-candidate metadata; no new compile-time check needed.
4. **F#16 (Testing as Proof)** — `previewGatedCount` is what makes the "preview.topK is tunable" guidance falsifiable: when the gate is biting, operators see it; when it isn't, the count is 0.

No backwards-compatibility shims. New fields are optional in the schema (`?` marker) so existing trace fixtures remain valid until they are regenerated by `145PREVCOMP-003`.

## What to Change

### 1. Per-candidate diagnostics extensions

In `agents/policy-diagnostics.ts`, extend the per-candidate metadata emission (around line 126-128 where `phase1Score` is emitted):

```ts
...(metadata.previewDriveDepth === undefined ? {} : { previewDriveDepth: metadata.previewDriveDepth }),
...(metadata.previewCompletionPolicy === undefined ? {} : { previewCompletionPolicy: metadata.previewCompletionPolicy }),
```

The driver returns these via the `DriveResult` (see `145PREVCOMP-001` D3); thread them through the preview-runtime to the diagnostics emitter. Use the same conditional-spread pattern as the existing Phase-1 fields.

### 2. Per-microturn gate diagnostics

Add `previewGatedCount` (number) at the microturn-trace level. The top-K gate in `policy-evaluation-core.ts` (`145PREVCOMP-002`) already knows the gated set; pass the count to the trace builder.

### 3. Optional flip-detection signal

Add `previewGatedTopFlipDetected?` (boolean) at the microturn-trace level. Emit only when:
- A gated candidate's outcome has materialized in the preview-runtime cache (e.g., from a subsequent evaluation pass within the same runtime instance).
- Re-scoring the gated candidate with its now-materialized preview outcome would yield a final score exceeding the chosen candidate's final score.

Sampling: do not force materialization. The check is opportunistic — if the cache happens to hold the gated outcome, compute the flip; otherwise emit nothing. This preserves the gate's cost guarantee per Spec 145 D7.

### 4. Schema updates

In `kernel/schemas-core.ts`:
- Per-candidate block (near line 1756): add `previewDriveDepth: NumberSchema.optional()`, `previewCompletionPolicy: z.enum(['greedy', 'agentGuided']).optional()`.
- Per-microturn block: add `previewGatedCount: NumberSchema.int().nonnegative().optional()`, `previewGatedTopFlipDetected: z.boolean().optional()`.

Mirror in `types-core.ts` for type-safety on consumers.

### 5. Reason-code emission

`'depthCap'` and `'gated'` and `'noPreviewDecision'` already flow through `previewFailureReason` strings from `145PREVCOMP-001` and `145PREVCOMP-002`. Verify the trace schema's `previewFailureReason` enum is open to new values (it is a string in current schemas) and that diagnostic output captures them. No new code path here, just an audit point.

### 6. `lessons-global.jsonl` entry

Append:

```jsonl
{"id": "spec-145-preview-ready-rate", "scope": "agent-preview", "lesson": "After Spec 145, expect previewOutcome: 'ready' rate above ~80% on full FITL games for non-stochastic action-selection candidates. Rates well below this in new campaigns suggest depthCap is too tight, topK is too tight, or a profile uses preview.mode: disabled — investigate before re-baselining."}
```

Adjust the lesson body to match existing JSONL entry conventions (check format of latest entry).

### 7. Diagnostic emission tests

- Driver emits `previewDriveDepth` and `previewCompletionPolicy` on every non-rejected candidate.
- Gate emits `previewGatedCount` matching the count of `'gated'` outcomes in the candidate set.
- Flip-detection: when a gated candidate's outcome IS cached and would flip ranking, the flag emits `true`; when it would not flip, no flag (or `false`); when the cache doesn't hold the outcome, no flag.
