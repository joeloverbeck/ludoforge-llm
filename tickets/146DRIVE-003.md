# 146DRIVE-003: Conformance witness suite for drive mutation safety, determinism, and shadow-canonical parity

**Status**: PENDING

**Deps**: `tickets/146DRIVE-002.md`

## Problem

Spec 146 Section D5 ("Mutation safety contract (F#11)") requires three architectural invariants for the new drive function:

1. **F#11 mutation safety**: caller-supplied `initialState` is never mutated; nested mutable fields preserved when no logical change occurred; the input's `stateHash` is unchanged after the call.
2. **Determinism**: same `(def, initialState, origin, depthCap, runtime)` produces identical `result.state.stateHash` across N independent invocations.
3. **Shadow-canonical parity** (Risks section mitigation): the shadow chain in `drive.ts` produces identical state hashes (after final canonicalization) to the canonical chain in `apply.ts` on a corpus of known transitions. This is the regression alarm for shadow-chain maintenance burden — if anyone diverges the two chains, this test fails.

All three invariants must be proven for both representative game families: FITL (decision-heavy, hidden information) and Texas Hold'em (stochastic, perfect information). This is the F#16 conformance corpus extension.

## Architecture Check

1. **Cleaner than alternatives**: A single witness suite covering all three invariants for both games avoids fragmenting the proof across multiple files. The `@witness` marker provides traceability back to the spec via grep.
2. **F#16 alignment (Testing as Proof)**: The conformance corpus gains a new architectural-invariant witness. Failures here are engine bugs and block CI per F#16's distinction (in contrast to `policy-profile-quality/` which emits non-blocking warnings).
3. **F#11 enforcement via test, not type-system**: F#11's exception clause says scoped mutation MUST be enforced by regression tests. This ticket *is* that enforcement layer. Static analysis cannot prove non-mutation across a 300-LoC shadow chain; runtime assertion against a production GameState corpus is the practical guard.
4. **Agnostic boundary preservation**: Tests parametrize on game family (FITL + Texas Hold'em) and assert *generic* properties (input non-mutation, hash determinism, parity vs canonical). No game-specific assertions about FITL phases or Texas hand rankings.

## What to Change

### 1. Create `packages/engine/test/determinism/spec-146-drive-mutation-safety-fitl.test.ts`

File-top markers:

```ts
// @test-class: architectural-invariant
// @witness: spec-146-drive-mutation-safety
```

Three test groups (`describe` blocks), each parametrized over a small fixture set (3-5 representative seeds covering early-game and mid-game decision states):

#### 1a. F#11 mutation safety

For each fixture state:

- Snapshot `initialState.stateHash` and a deep snapshot of nested mutable fields (`decisionStack`, `players`, zone token bags, marker maps). Use a structured-clone deep equal helper or a JSON-serialized snapshot.
- Call `applyPreviewDriveGreedyChooseOne(def, initialState, origin, K_PREVIEW_DEPTH, runtime)`.
- Assert `initialState.stateHash === <pre-call snapshot hash>` (the stored hash on the input object is untouched).
- Assert `computeFullHash(table, initialState) === <pre-call computed hash>` (the input's actual content is untouched — distinct from the stored hash).
- Assert nested mutable fields deep-equal their snapshots.

Per spec D5: do NOT assert `Object.is(initialState, returnedState) === false`. Early exits at depth 0 may legitimately return the input reference unchanged.

#### 1b. Determinism (N independent invocations)

For each fixture state, invoke `applyPreviewDriveGreedyChooseOne` N=10 times with identical inputs. Assert all 10 results have identical `state.stateHash`, `depth`, and `kind`. This proves the new function does not introduce non-determinism (Foundation 8).

#### 1c. Shadow-canonical parity

For each fixture state, build a corpus of 8-12 transitions:

- Loop side: invoke `applyPreviewDriveGreedyChooseOne(def, initialState, origin, depthCap=8, runtime)` and capture `result.state.stateHash`.
- Reference side: replicate the same loop manually using `applyPublishedDecisionFromCanonicalState` (canonical, hashing) per iteration with the same greedy-pick logic. Capture the final `stateHash`.
- Assert `loop.state.stateHash === reference.state.stateHash`. This proves the shadow chain produces an identical canonical state to the canonical chain — the regression alarm for shadow-chain maintenance burden.

### 2. Create `packages/engine/test/determinism/spec-146-drive-mutation-safety-texas.test.ts`

Same three test groups as 1a/1b/1c, parametrized for Texas Hold'em fixture states. Texas's stochastic decisions are out-of-scope for the drive function (it exits with `kind: 'stochastic'` on stochasticResolve top frames), so most fixtures should be pre-flop or post-flop chooseOne sequences. Confirm via test setup that at least 2 fixtures actually exercise the inner-loop iteration path (depth >= 2).

### 3. Optional: shared test helpers

If the FITL and Texas test files share a non-trivial fixture-construction or loop-replay helper, extract to `packages/engine/test/helpers/drive-parity-helpers.ts`. Keep the helper game-agnostic — accept `def`, `initialState`, `origin`, `depthCap` and return the comparison result. Do NOT add game-specific branches in the helper.

### Invariants

1. **F#11 mutation safety**: For every fixture and every drive invocation, `initialState`'s stored hash, content hash, and nested mutable fields are unchanged after the call.
2. **Determinism (Foundation 8)**: 10 independent invocations of the drive on identical inputs produce identical `state.stateHash`, `depth`, and `kind`.
3. **Shadow-canonical parity**: For every fixture in the corpus, the drive function's final `state.stateHash` equals the canonical-chain reference path's final `state.stateHash`. Any divergence is a shadow-chain regression and must fail loudly.
4. **Witness ID present**: Both new test files have `// @witness: spec-146-drive-mutation-safety` as the second comment line. Verifiable by grep.
5. **Test class invariant**: Both new test files have `// @test-class: architectural-invariant` as the first comment line.
