# 146DRIVE-004: Migrate `driveSyntheticCompletion` greedy-chooseOne path + perf gate validation

**Status**: PENDING
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — `packages/engine/src/agents/policy-preview.ts`
**Deps**: `tickets/146DRIVE-002.md`

## Problem

Spec 146 closes the FITL preview-pipeline performance gap by replacing the per-iteration greedy-chooseOne loop in `driveSyntheticCompletion` with a single call to the new kernel-internal `applyPreviewDriveGreedyChooseOne`. The migration is the load-bearing step — it's where the performance benefit is actually realized.

Per the spec's overall acceptance criteria (#1, #5), this ticket also runs the performance gate: the FITL preview-pipeline perf test must drop from ~31s post-campaign to ≤25.6s (the spec's hard target).

## Assumption Reassessment (2026-04-26)

1. The current greedy-chooseOne fast path in `policy-preview.ts:758-765` gates on `ctxKind === 'chooseOne' && completionPolicy === 'greedy'`. The migration target is exactly these 8 lines. Confirmed during reassessment.
2. The cheap state-stack exit checks at `policy-preview.ts:732-752` (introduced by exp-014) are duplicated by spec D2 inside `applyPreviewDriveGreedyChooseOne`. After migration, these checks remain in `policy-preview.ts` for the ENCLOSING loop (which still handles the slow path for `chooseNStep` and `agentGuided`); the new function performs its own internal checks for its inner-loop iterations.
3. The slow-path branches at `policy-preview.ts:768-774` (chooseNStep + non-greedy chooseOne) MUST remain unchanged — spec D4 explicitly preserves them.
4. `K_PREVIEW_DEPTH = 8` at `policy-preview.ts:41` remains the depth cap; pass it as the `depthCap` argument to the new function.
5. `DriveResult` union at `policy-preview.ts:198-221` (`completed | stochastic | depthCap | failed`) is structurally identical to `PreviewDriveResult` (introduced by 146DRIVE-001). The mapping is direct field-for-field.
6. `applyPreviewDriveGreedyChooseOne` is exported from `kernel/microturn/drive.ts` (delivered by 146DRIVE-002).
7. `packages/engine/test/perf/agents/preview-pipeline.perf.test.ts` exists and is the perf gate.
8. The conformance witness suite from 146DRIVE-003 is not a hard prerequisite for implementation but should be passing before merge — value dependency, not blocking.

## Architecture Check

1. **Cleaner than alternatives**: Single-line replacement in the caller. The new function owns the loop; the caller owns the policy gating (`completionPolicy === 'greedy'`). Spec D4's surface-area guidance is honored — only the gated chooseOne+greedy path is migrated.
2. **F#11 contract preserved at the call site**: The caller continues to pass canonical state (it had `state.stateHash` canonical from the previous iteration's `applyPublishedDecisionFromCanonicalState` return). The new function's input contract matches this.
3. **Foundation 5 (One Rules Protocol)**: The new function is a kernel API; the agent driver still consults the kernel for every decision via the publish/apply pipeline (now wrapped behind one entry point). No bespoke agent rule logic introduced.
4. **Foundation 8 (Determinism) preserved**: Final state hash is canonical and identical to the per-iteration immutable path's output (proven by the parity test in 146DRIVE-003).
5. **Spec deliverable folding rationale**: The spec's Follow-On Tickets section listed item 5 as "Run the perf test". Folding this into 004's acceptance criteria rather than creating a separate ticket recognizes that the perf validation IS the acceptance gate for the migration, not a separable work artifact. A failed perf gate would be a regression of THIS ticket, not a downstream investigation.

## What to Change

### 1. Replace the greedy-chooseOne fast path block in `policy-preview.ts`

Locate the current block at `policy-preview.ts:758-765`:

```ts
if (ctxKind === 'chooseOne' && completionPolicy === 'greedy') {
  const greedy = publishMicroturnGreedyChooseOne(input.def, state, input.runtime);
  if (greedy === null) {
    return { kind: 'failed', reason: 'noPreviewDecision', depth, failureReason: 'noPreviewDecision' };
  }
  state = applyPublishedDecisionFromCanonicalState(input.def, state, greedy.microturn, greedy.decision, { advanceToDecisionPoint: true }, input.runtime).state;
  depth += 1;
  continue;
}
```

Replace with a single call to the new function. The replacement block:

1. Constructs `origin: PreviewDriveOrigin = { seatId: origin.seatId, turnId: <current top.turnId or origin.turnId> }`. The existing `origin` variable in scope at line 704 is from `publishMicroturnFromCanonicalState` and has both fields; reuse it.
2. Calls `applyPreviewDriveGreedyChooseOne(input.def, state, origin, K_PREVIEW_DEPTH - depth, input.runtime)`. Note: pass `K_PREVIEW_DEPTH - depth` so the new function's internal cap accounts for iterations already completed by the trustedMove apply at line 709-723. Confirm the math during implementation: if the current loop has `depth = 1` after the trustedMove, the remaining budget is `K_PREVIEW_DEPTH - 1 = 7`.
3. Maps `PreviewDriveResult` to the enclosing function's return shape:
   - `kind: 'completed'` → `return { kind: 'completed', state: result.state, depth: depth + result.depth };`
   - `kind: 'stochastic'` → `return { kind: 'stochastic', state: result.state, depth: depth + result.depth };`
   - `kind: 'depthCap'` → `return { kind: 'depthCap', state: result.state, depth: depth + result.depth };`
   - `kind: 'failed'` → `return { kind: 'failed', reason: 'noPreviewDecision', depth: depth + result.depth, failureReason: result.failureReason ?? 'noPreviewDecision' };`

The slow-path branch at lines 768-774 remains unchanged.

### Invariants

1. **Slow-path preservation**: The chooseNStep/agentGuided branches at `policy-preview.ts:768-774` are byte-identical pre/post migration. Verifiable by `git diff -U0`.
2. **K_PREVIEW_DEPTH semantics preserved**: Total drive depth (counting the trustedMove apply + inner-loop iterations) never exceeds `K_PREVIEW_DEPTH = 8`.
3. **DriveResult contract preserved**: Every code path in `tryApplyPreview` returns a `DriveResult` with `kind ∈ {completed, stochastic, depthCap, failed}`, and the `depth` field is monotonically non-decreasing across the migration.
4. **Foundation 11 input non-mutation at call site**: The `state` variable passed into `applyPreviewDriveGreedyChooseOne` is reassigned to `result.state` (the canonical output), not mutated in place. The drive function's internal mutation discipline is enforced by 146DRIVE-003's tests.
