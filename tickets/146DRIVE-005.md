# 146DRIVE-005: Investigate post-migration preview perf shortfall

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — likely `packages/engine/src/agents/*` and/or follow-on spec/ticket updates after measurement
**Deps**: `archive/tickets/146DRIVE-004.md`, `specs/146-scoped-draft-state-for-preview-drive.md`, `specs/147-aot-consideration-ast-compilation.md`

## Problem

146DRIVE-004 successfully migrated the `driveSyntheticCompletion` greedy `chooseOne` branch to `applyPreviewDriveGreedyChooseOne`, but the measured FITL preview-pipeline corpus still misses Spec 146's hard target.

Post-migration direct measurement on 2026-04-26:

- `run1_totalMs=28461.51`
- `run2_totalMs=27425.41`
- `run3_totalMs=27888.67`
- `mean_totalMs=27925.19`
- `mad_pct=1.28`
- `hardTargetMs=25600`
- `pass=false`

The result is meaningfully faster than the post-campaign ~31149 ms baseline but remains about 2325 ms above the hard target. Spec 147 already proposes AOT consideration AST compilation as a likely next optimization, but this ticket must first classify the remaining gap with fresh profile evidence rather than assuming the remaining cost belongs entirely to Spec 147.

## Assumption Reassessment (2026-04-26)

1. `packages/engine/src/agents/policy-preview.ts` now calls `applyPreviewDriveGreedyChooseOne` on the gated greedy `chooseOne` path; the migration itself is landed by 146DRIVE-004.
2. `packages/engine/test/perf/agents/preview-pipeline.perf.test.ts` still emits only the older non-blocking baseline warning. It does not assert the Spec 146 hard target or print `totalMs` when the warning threshold is not exceeded.
3. The hard-target verdict therefore needs either a dedicated harness/report path or a test update before future closeout can rely on the named perf lane alone.
4. The broad determinism lane timed out in `dist/test/determinism/zobrist-incremental-parity.test.js` after 20 minutes during 146DRIVE-004 verification, after both Spec 146 determinism witnesses had passed. That timeout must be classified as pre-existing lane cost, post-migration slowdown, or separate runner threshold drift before closing the remaining performance series.

3. If Spec 147 is confirmed as the correct owner, this ticket should create or update the first Spec 147 implementation ticket rather than duplicating that design under 146DRIVE.
