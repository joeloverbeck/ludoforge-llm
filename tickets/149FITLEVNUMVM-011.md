# 149FITLEVNUMVM-011: Bytecode opcode set + IR types + PolicyBytecode schema

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — new module `packages/engine/src/cnl/policy-bytecode/`
**Deps**: `specs/149-fitl-evolution-readiness-numeric-substrate-bytecode-vm.md`

## Problem

Phase 3 of spec 149 introduces a flat numeric bytecode IR for policy expressions. This ticket lands the foundational types: the opcode enumeration, instruction shape, feature-id table type, and the JSON Schema for the compiled artifact. No compiler logic yet — that's ticket 013.

## Assumption Reassessment (2026-04-28)

1. The compile-time `AgentPolicyExpr` IR exists in `packages/engine/src/kernel/types.ts` (referenced by `policy-eval.ts` and `compiled-policy-runtime.ts`). The bytecode opcode set must cover every variant of `AgentPolicyExpr` and `CompiledPolicyExpr` (zoneTokenAgg, globalTokenAgg, globalZoneAgg, adjacentTokenAgg, seatAgg, etc.).
2. No existing `PolicyBytecode` type or schema in the codebase (verified via Glob during spec 149 reassessment).
3. The schema artifact lives in `packages/engine/schemas/PolicyBytecode.schema.json` per spec §Phase 3. The existing schema directory is at that path.

## Architecture Check

1. Opcodes are game-agnostic — FITL-specific operators (e.g., "support shift", "trail chain") decompose into compositions of generic opcodes via the existing macro/AST layer (verified — see `expand-effect-macros.ts`, `expand-condition-macros.ts`). F1 (Engine Agnosticism) preserved.
2. The schema is a single generic schema, not per-game. F6 (Schema Ownership Stays Generic) preserved.
3. Bytecode is generated by the compiler (ticket 013), not authored. F7 (Specs Are Data) preserved — GameSpecDoc remains declarative YAML.
4. Integer-only opcodes; no floats. F8 (Determinism) preserved — exact arithmetic.

## What to Change

### 1. `packages/engine/src/cnl/policy-bytecode/types.ts` (new)

Export:
- `enum Opcode` (or const-enum) per spec §2.4: `LOAD_FEATURE`, `LOAD_CONST`, `GT`, `LT`, `EQ`, `NEQ`, `JUMP_IF_FALSE`, `ADD_SCORE`, `MUL_SCORE`, `RESOLVE_REF`, `AGGREGATE_SUM`, `AGGREGATE_COUNT`, `AGGREGATE_MIN`, `AGGREGATE_MAX`, `RESOLVE_DYNAMIC` (transitional fallback per spec §5), `HALT`.
- `interface BytecodeInstruction` — operand shape per opcode.
- `interface PolicyBytecode` — top-level shape: `{ instructions: Int32Array; featureTable: FeatureTable; constants: Int32Array; metadata: BytecodeMetadata }`.
- `interface BytecodeMetadata` — version, source `AgentPolicyExpr` fingerprint, target VM version.
- `interface FeatureTable` (forward-declared; concrete impl lands in ticket 012).

### 2. `packages/engine/schemas/PolicyBytecode.schema.json` (new)

JSON Schema describing the `PolicyBytecode` artifact for compile-time validation. Use Draft 2020-12 (matches existing schemas in the directory).

### 3. Score-overflow guard scaffolding

Spec §11 open question 2: Int32 vs BigInt. Default to Int32 with a compile-time range analysis hook (a function stub `validateScoreRange(expr: AgentPolicyExpr): RangeAnalysis` — full implementation lands in ticket 013 with the compiler). Cite the spec §5 edge case "score math is Int32 (with overflow guard at compile time — bytecode compiler refuses expressions whose static range exceeds 2^30)".

### 4. Disassembler ABI hook

Reserve a `disassemble(bytecode: PolicyBytecode): string` function signature in this ticket's exports (concrete impl in ticket 013). This lets ticket 013 land both compiler and disassembler in one ticket without churning ticket 011's exports.
