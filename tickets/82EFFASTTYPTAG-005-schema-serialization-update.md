# 82EFFASTTYPTAG-005: Schema Serialization Update for _k Field

**Status**: PENDING
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — `packages/engine/src/kernel/schemas-ast.ts`, schema artifacts
**Deps**: 82EFFASTTYPTAG-001 (needs `EFFECT_KIND_TAG`)

## Problem

The Zod schemas in `schemas-ast.ts` and the JSON Schema artifacts in
`packages/engine/schemas/` do not include the `_k` field. Per Foundation 5
(round-trip integrity) and Foundation 9 (no backwards-compat shims), `_k`
must be serialized in GameDef JSON as a required integer field. The schema
must validate its presence.

## Assumption Reassessment (2026-03-25)

1. `schemas-ast.ts` defines `effectAstSchemaInternal` as a Zod union of 34
   effect schemas (lines 539-941). Each schema wraps the payload in its kind
   key. Confirmed.
2. The schema is lazy-initialized via `EffectASTSchema` (line 123) to support
   recursion. Confirmed.
3. JSON Schema artifacts are generated by `pnpm turbo schema:artifacts` and
   live in `packages/engine/schemas/`. Confirmed.
4. `assertValidatedGameDef` uses these schemas for validation. Confirmed.

## Architecture Check

1. Adding `_k` as a required `z.number().int()` field to each of the 34
   effect schema entries is the correct approach — it mirrors the type-level
   change from ticket 001.
2. Per Foundation 9, `_k` is required — no optional fallback.
3. Schema artifacts must be regenerated after the Zod schema update.

## What to Change

### 1. Update `schemas-ast.ts`

For each of the 34 effect schema entries in `effectAstSchemaInternal`, add
`_k: z.number().int()` as a required field. This can be done by intersecting
each entry with a `_k` field, or by adding `_k` to each `z.object()` call.

The cleanest approach: define a base schema fragment
`const effectTagSchema = z.object({ _k: z.number().int() })` and intersect
it with each effect variant schema.

### 2. Regenerate JSON Schema artifacts

Run `pnpm turbo schema:artifacts` to update the generated JSON Schema files
in `packages/engine/schemas/`.
