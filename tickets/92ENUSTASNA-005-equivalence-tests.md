## Problem

The spec's acceptance criterion #3 requires proving that compiled-with-snapshot evaluation produces identical results to compiled-without-snapshot evaluation for ALL pipeline predicates across N random states. This is the correctness proof that the snapshot optimization is safe.

5. Random state generation can be done by applying N random moves from `legalMoves` with different seeds.

## What to Change

### 1. Equivalence test: per-predicate comparison

For each compiled pipeline predicate in the FITL GameDef:
- Evaluate WITH snapshot: `compiled(state, activePlayer, bindings, snapshot)`
- Evaluate WITHOUT snapshot: `compiled(state, activePlayer, bindings, undefined)`
- Assert both return the same boolean value

Repeat across N game states (at least 10) generated by applying random legal moves from different seeds.

At least part of the matrix must evaluate predicates under an execution player different from `state.activePlayer` so the generalized per-player snapshot access is proven directly.

### 2. Equivalence test: full legalMoves output comparison

For N game states:
- Call `legalMoves(def, state)` (which internally creates and uses the snapshot)
- Call a reference implementation that evaluates predicates without snapshot
- Assert the move sets are identical (same move IDs, same classification)

This can be done by comparing the legalMoves output against a baseline captured before the snapshot wiring (golden output approach), or by temporarily removing the snapshot and comparing (dual-call approach).

**Recommended approach**: Dual-call — for each state and evaluation-player context, call each compiled predicate twice (once with snapshot, once without) and assert equality. This is more robust than golden outputs because it tests the actual composition.
